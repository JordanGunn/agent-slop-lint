# 04 — composition.* collapse and v1.2 rule rollout

**Status:** backlog (not scheduled)
**Target:** v1.2.0
**Predecessor docs:** [observation 03](../observations/composition/03.md),
[naming spec](../philosophy/naming-spec.md), per-method docs in
[`docs/methods/composition/`](../methods/composition/).

## Context

The v1.1.0 release is held on `feature/v1.1.0-composition-and-lexical`
and not tagged. During the dogfood and round-4 PoC research, four
findings converged:

1. The single-signal `composition.first_parameter_drift` rule
   produces ~25% actionable findings (observation 03). The fix is
   multi-signal detection, not threshold tuning.
2. Multiple distinct review actions (extract class, split file,
   model alphabet as type, rename for consistency) hide inside
   the single `composition.first_parameter_drift` advisory.
3. The `composition.*` suite name violates slop's own
   substrate-naming convention.
4. The rule names accumulated without scrutiny — the
   meta-irony recorded in the naming spec.

This plan resolves all four together as a single v1.2.0 release.

## Final rule layout

`composition.*` ceases to exist as a suite. Five rules collapse
into `lexical.*` under spec-conformant names. Three v1.1.0 rules
are renamed to comply with the audit findings, and the v1.1.0
stutter split is unified back into a single hierarchy-aware rule.

| New name | Replaces | Status |
|---|---|---|
| `lexical.sprawl.scoped` | `composition.affix_polymorphism` (within-scope) | rename + minor kernel cleanup |
| `lexical.sprawl.global` | (new — was proposed as `composition.implicit_enum`) | new rule; port PoC v2.3 |
| `lexical.imposters` | `composition.first_parameter_drift` | rename + kernel upgrade (multi-signal) |
| `lexical.slackers` | (new — was proposed as `composition.sibling_drift`) | new rule; port PoC v2.4 |
| `lexical.confusion` | (new — was proposed as `composition.multi_concern_module`) | new rule; port PoC v2.7 |
| `lexical.cowards` | `lexical.numbered_variants` | rename — current name is misleadingly narrow (rule covers `_old`, `_new`, `_local` etc., not just numbered) |
| `lexical.singletons` | `lexical.identifier_singletons` | rename — "identifier" is redundant under the lexical suite |
| `lexical.tautology` | `lexical.type_tag_suffixes` | rename — token-economy violation; new name names the smell directly |
| `lexical.stutter` | `lexical.stutter.{namespaces, callers, identifiers}` | unification — kernel becomes hierarchy-aware; per-level toggle via parameters |

## Method-to-rule binding

The seven methods documented in
[`docs/methods/composition/`](../methods/composition/) feed the
five rules as follows:

| Rule | Primary method(s) | Supporting method(s) |
|---|---|---|
| `lexical.sprawl.scoped` | within_cluster_affix (v2.4) detecting closed alphabet | dominant_text (v2.1) for labeling |
| `lexical.sprawl.global` | closed_alphabet_entity (v2.3) | dominant_text (v2.1) for labeling |
| `lexical.imposters` | multi_criteria_ranking (v2.6) | body_shape_signatures (v2.2) as strategy disambiguator |
| `lexical.slackers` | within_cluster_affix (v2.4) detecting *low* coverage on a real cluster | dominant_text (v2.1) for labeling |
| `lexical.confusion` | lanza_marinescu_detection (v2.7) | first-param clustering primitive |

`latent_topic_modeling` (v2.5) is not promoted to any rule. Retained
for orientation research only, as documented in its method page.

### Phase 1b — Stutter unification (~half day)

The v1.1.0 split into `lexical.stutter.{namespaces, callers,
identifiers}` was a workaround for the unified rule firing
noisily; the right fix is to make the kernel hierarchy-aware so
findings are properly attributed to the level they occurred at,
without splitting the rule.

- Collapse the three sub-rules back into one `lexical.stutter`
- Kernel walks every named entity and checks against ALL
  enclosing scopes (package → module → class → function)
- Per-finding metadata records which level the stutter occurred
  at — `module`, `class`, `function`, or `package`
- Configuration parameters dial down specific levels:
  ```toml
  [rules.lexical.stutter]
  check_packages = true
  check_modules = true
  check_classes = true
  check_functions = true
  min_overlap_tokens = 2
  ```
- Catches a case the v1.1.0 split missed: method NAMES stuttering
  with their class name (e.g., `class UserService: def
  get_user_service_id(...)` — the method name itself stutters with
  the class). The split rules check identifiers inside method
  bodies but not the method names themselves.

Touch files:
- `src/cli/slop/_lexical/stutter.py` — kernel becomes hierarchy-
  aware; existing `scope_filter` parameter generalized
- `src/cli/slop/rules/stutter.py` — collapse three wrappers into
  one
- `src/cli/slop/rules/__init__.py` — registry: replace three
  entries with one
- `src/cli/slop/config.py` — `DEFAULT_RULE_CONFIGS` for unified
  rule
- `src/cli/slop/_compat.py` — translate legacy
  `[rules.lexical.stutter.namespaces]` etc. to the unified rule's
  per-level toggles
- `src/tests/test_rules/test_stutter.py` — update for unified
  rule; add tests for method-name-stutter case
- `docs/rules/lexical/stutter.md` — rewrite for unified design

### Phase 2 — `lexical.imposters` kernel upgrade (~1 day)

The existing kernel does single-signal first-param clustering.
Multi-signal upgrade per observation 03:

- Integrate v2.6 multi-criteria ranking: per-member body-shape
  similarity, receiver-call density, modal-token overlap.
- Integrate v2.2 body-shape signatures as strategy disambiguator:
  high body-shape Jaccard + zero receiver-calls = strategy family
  (suppress refactor advisory).
- Verdict classifier upgrades from strong/weak/false-positive to
  signal-profile reporting. Each cluster reports the four
  measurements; the rule docs explain how to read them.
- The rule still emits violations for strong-receiver clusters,
  but the message includes the signal profile so the reader sees
  *why*.

Touch files:
- `src/cli/slop/_lexical/imposters.py` (new — extracted from
  `_structural/composition.py`)
- `src/cli/slop/rules/imposters.py` (new wrapper)
- Tests: 5-8 new cases covering signal-profile reporting.

### Phase 3 — `lexical.slackers` (new rule; ~half day)

Depends on Phase 2 (consumes the same first-parameter clustering
primitive).

- Port v2.4 within-cluster affix detection.
- Rule fires on clusters with low affix coverage (< 30%) where
  the cluster size is ≥ minimum threshold.
- Requires a "real cluster" predicate — the cluster must be a
  strong candidate per the imposters multi-criteria ranking.
  Otherwise we'd flag any random parameter-name coincidence.

Touch files:
- `src/cli/slop/_lexical/slackers.py` (new)
- `src/cli/slop/rules/slackers.py` (new wrapper)
- Tests.

### Phase 4 — `lexical.sprawl.global` (new rule; ~1 day)

Independent of Phases 2-3.

- Port v2.3 closed-alphabet entity recognition.
- Walks the corpus for affix patterns, scores each closed alphabet
  by stems × files × module-dict-key matches, reports high-scoring
  alphabets.
- Module-level dict-key detection requires AST walking of
  module-level assignments (already prototyped in PoC).

Touch files:
- `src/cli/slop/_lexical/sprawl.py` — extend with global variant
  (or split into `sprawl_scoped.py` + `sprawl_global.py`)
- `src/cli/slop/rules/sprawl.py` — register both sub-rules
- Tests.

### Phase 5 — `lexical.confusion` (new rule; ~1 day)

Independent of Phases 2-4.

- Port v2.7 Lanza/Marinescu detection.
- File-level rule: counts functions per file, counts distinct
  strong-receiver clusters per file, applies the AND'd threshold
  rule.
- Reuses the imposters cluster classification from Phase 2.

Touch files:
- `src/cli/slop/_lexical/confusion.py` (new)
- `src/cli/slop/rules/confusion.py` (new wrapper)
- Tests.

## Compat layer

Legacy names get translated automatically via `slop._compat`:

```python
LEGACY_RULE_NAMES.update({
    "composition.affix_polymorphism": "lexical.sprawl.scoped",
    "composition.first_parameter_drift": "lexical.imposters",
    "lexical.type_tag_suffixes": "lexical.tautology",
    "lexical.numbered_variants": "lexical.cowards",
    "lexical.identifier_singletons": "lexical.singletons",
})
```

TOML table migrations: `[rules.composition.affix_polymorphism]` →
`[rules.lexical.sprawl.scoped]` etc.

The compat layer is cheap insurance. The composition.* rules were
on a feature branch and never tagged, so in principle no user
config could reference them. But the tests, the dogfood
`.slop.toml`, and any local development branches do reference the
old names — the compat shim lets those keep working as-is during
the transition.

Removal in v1.3.0 (per the existing `_compat.py` deprecation
contract).

## Calibration plan

Each new rule has thresholds picked for slop's corpus only. Before
shipping v1.2.0:

1. **Run the v2 PoC battery on 5-10 diverse Python codebases.**
   Recommendations: a Django app, a Flask service, a numpy/scipy
   data-science codebase, a CLI tool, a typed library, an
   agent-written codebase the user has access to.
2. **Record per-rule findings per corpus.** Tabulate: total
   findings, manually-reviewed actionable rate, false-positive
   shape.
3. **Adjust thresholds** to land on >=70% actionable across the
   corpus average. The current PoC numbers are first-pass; cross-
   corpus calibration is required for production thresholds.
4. **Cross-language verification.** Re-run on JS/TS/Java/Ruby
   corpora. Tree-sitter abstracts most surface differences but
   per-language receiver-call heuristics need verification.
5. **Findings recorded** as observation 04 alongside any threshold
   updates.

Calibration is a deliberate gate before v1.2.0 ships.

## Ship strategy

Two paths:

**(a) Amend v1.1.0** — rewrite the existing v1.1.0 commit on the
feature branch to use the new names from the start. The
`composition.*` suite never reaches users. v1.2.0 then layers the
new rules on top.

**(b) Ship v1.1.0 as-is, then v1.2.0** — release v1.1.0 with
`composition.*`, then release v1.2.0 a week later with the rename
and new rules.

I'd recommend **(a)**. Reasons:

- Nothing is tagged. v1.1.0 has no users.
- Shipping `composition.*` only to deprecate it the next release
  trains users to expect churn.
- The naming-spec discussion happened *because* the names were
  wrong; shipping the wrong names contradicts the spec we just
  wrote.
- The amend is cheap (no API users, only test fixtures and dogfood
  config).

Path (a) means: rebase the v1.1.0 commit to use new names; commit
the new rules and kernel upgrade on top; tag the result as v1.2.0
(skip v1.1.0 number entirely or use v1.1.0 for an internal
checkpoint that doesn't go to PyPI).

## Out of scope (for v1.2.0)

- **Cross-rule correlation** (the "this name_verbosity finding IS
  the same cluster as imposters" insight from observation 03).
  Worth a v1.3.0 follow-up; presentation concern, not detection.
- **`lexical.sprawl.compounding`** (sprawl that ladders across
  parent-child scopes). Worth a v1.3.0 follow-up; needs the v1.2
  base sprawl detection to validate first.
- **Output compression** (backlog/03.md). Independent; can ship
  before, after, or alongside v1.2.0.
- **RTM proper** (Bavota Methodbook with relational signal). The
  PoC uses simplified LDA; full RTM with call-graph edges is a
  research-grade addition, deferred indefinitely until we see a
  case the simpler methods can't handle.

## Risks

- **Calibration takes longer than expected.** The cross-corpus
  threshold tuning could surface failure modes the slop-only
  testing missed. Mitigation: budget calibration as elapsed-time,
  not focused-effort, and treat threshold adjustments as the work
  rather than a gate.
- **Some new rules may not produce useful signal on real
  corpora.** PoC findings on slop are encouraging but slop is one
  data point. If `lexical.sprawl.global` produces only the
  Language alphabet on slop and nothing on a Django app, that's
  the rule failing to generalize. Mitigation: ship rules that
  pass calibration; defer rules that don't to v1.3 with more
  research time.
- **The compat shim doesn't catch every legacy reference.** If a
  user has a script that reads slop's JSON output and looks for
  `composition.*` rule names, the compat shim doesn't help them.
  Mitigation: CHANGELOG flagging, JSON output format kept stable
  (rules just appear with new names).

# 08 — Profile matching and diagnostic output: the v2.0 capstone

**Status:** horizon (v2.0+ research) — depends on prior phases
**Predecessor:** [05.md](05.md), [06.md](06.md), [07.md](07.md)
**Successor:** v2.x cross-language extension; named-fix advice library

## Context

Phase 6 produces fingerprints — labelled profiles for known
smells. Phase 7 produces profiles — structured summaries of rule
outputs at each scope. Phase 8 connects them: given a real-code
profile, find which fingerprint it matches, and return a named
diagnostic with actionable refactor advice.

This is the move from "rule X fired" to "this looks like Feature
Envy; here's what that means; here's how to act."

## What this changes for the user

Three output modes:

1. **`slop lint`** (existing) — per-rule violations, terminal
   formatted. Stays as-is for users who want raw findings.
2. **`slop check <rule>`** (existing) — single-rule output.
3. **`slop diagnose`** (new) — profile-matched output. Each
   finding is a named smell with a confidence score, the matching
   profile slice, and a link to refactor guidance.

Sample `slop diagnose` output:

```
slop 2.0 diagnose — scanning /repo/billing

cli/billing/customer_ops.py
  → Feature Envy (confidence 0.82)
    The cluster of 5 functions on `customer` accesses customer's
    attributes 4.4 times per member on average. These functions
    are de-facto methods on Customer.
    Suggested refactor: Move Method (extract the cluster as
    methods on the Customer class).
    Reference: refactoring.guru/feature-envy

  → Primitive Obsession (confidence 0.61)
    Closed alphabet of 6 type-tag suffixes (_id, _str, _dict,
    _path) recurs across 18 functions. The codebase is using
    primitives where named types would clarify intent.
    Suggested refactor: Replace Primitive with Object.
    Reference: refactoring.guru/primitive-obsession
```

Compared with `slop lint` output for the same code, which would
say:

```
lexical.imposters
  ⚠ output.py:50 customer — 5 functions in `output.py` share `customer`
    as first parameter; receiver-call density 4.4 per member, ...
lexical.tautology
  ⚠ ... 18 violations of _id, _str, _dict, _path suffixes ...
```

The `diagnose` output is *prescriptive* and *named*; the `lint`
output is *descriptive* and *raw*. Both have their place.

## Algorithm

```
profiles = collect_profiles(root, config)        # Phase 7
fingerprints = load_fingerprints(catalog_dir)    # Phase 6 corpus

for scope, profile in profiles.items():
    matches = []
    for smell, fingerprint in fingerprints.items():
        distance = profile_distance(profile, fingerprint)
        if distance < threshold:
            confidence = 1.0 - distance / threshold
            matches.append((smell, confidence))
    matches.sort(key=lambda m: -m[1])
    if matches:
        report(scope, matches)
```

The interesting work is in `profile_distance`. Several
candidates:

### Cosine distance over metric vectors

Treat each profile as a vector in metric space; compute cosine
similarity. Pros: standard, well-understood. Cons: ignores
structural cross-refs (two rules firing on the same artifact is
different from firing on different artifacts).

### Weighted Manhattan with rule-specific weights

Each rule has a weight in the matching, calibrated against the
fingerprint corpus. Pros: lets us tune which rules matter most
for which smell. Cons: weight calibration is its own subproblem.

### Boolean-pattern + numeric refinement

Two-phase: first match the *fired-rule pattern* (which rules
fired, ignoring magnitudes); among matches, rank by metric-value
distance. Pros: robust to scale, fast. Cons: might miss weak
matches where a key rule fired below threshold.

### Tree edit-distance over scope structure

Profiles have hierarchy (cluster → file → package). Smells too
(e.g., Large Class is file-scope; Feature Envy is cluster-scope).
Match at the right level. Pros: respects scope. Cons: most
complex.

I'd start with **weighted Manhattan + threshold-tuned**. Iterate
based on what the fixture corpus reveals.

## Confidence calibration

Confidence reporting requires care. Two failure modes:

1. **Overconfident matches.** A profile resembles Feature Envy
   on 3 of 9 rules; we report 0.82 confidence; user trusts it
   and refactors; the actual smell was Inappropriate Intimacy
   (which shares those 3 rules).
2. **Underconfident matches.** A real Feature Envy fires only
   imposters; we report 0.45 confidence because other rules
   "should" have also fired; user dismisses the finding.

Mitigations:

- **Calibration corpus.** Run the fixture corpus through matching;
  measure precision/recall; tune the distance threshold.
- **Multi-match output.** Don't pick a single smell; list the
  top 3 candidates with their confidences. Reader picks.
- **Rule-firing required-for-smell list.** For each smell, mark
  which rules MUST fire for the match to be valid (Feature Envy
  requires imposters; without it, no match regardless of other
  rules).

## Output format design

Two outputs to consider:

### Terminal-readable (`--output diagnose`)

Per-scope: top 1-3 candidate smells, confidence, summary of why,
suggested refactor name with literature link. Compact, oriented
toward review.

### Machine-readable (`--output diagnose-json`)

Full profile + match details + alternative matches. For
integration into editor / CI tools.

## Relationship to refactoring.guru-style guidance

Each named smell has a corresponding refactor (or family). For
v2.0:

- Smell name → refactor name (Move Method, Extract Class,
  Replace Primitive with Object, etc.)
- Refactor name → external link (refactoring.guru,
  Fowler 2nd ed. chapter)
- Inline summary (3-5 lines) — what the refactor does, when to
  reach for it, when not to.

Long-term (v3.0+ territory): some refactors are mechanical
enough that slop could *suggest the diff*. Move Method is a
candidate. That's far out; v2.0 stops at named recommendation
+ external reference.

## Failure modes to handle

- **Profile matches no fingerprint.** Don't say "no smell" — say
  "no match in catalog." Code might be fine, or it might be a
  novel pattern. Distinguish.
- **Profile matches multiple fingerprints with similar
  confidence.** Either composite smell or genuinely ambiguous.
  Report both.
- **Profile is empty (no rules fired).** File is clean; quiet
  output.
- **Smell fingerprint catalog is incomplete.** Most likely state
  for v2.0 — we're shipping with 12-15 smells. Document
  explicitly: "slop's catalog covers <list>; novel patterns may
  not be diagnosed."

## Implementation order

1. **Distance metric.** Implement weighted Manhattan over profile
   slices. Tune weights against fixture corpus.
2. **Matcher.** Apply distance metric to all (profile, fingerprint)
   pairs; rank.
3. **Output mode.** Wire into CLI; format diagnose output.
4. **Confidence calibration.** Run fixture corpus end-to-end;
   tune threshold; write calibration into observation 05+.
5. **Refactor advice library.** Inline summaries per smell with
   external references.

## Effort

- Distance metric + matcher: 2 days
- Output mode (terminal + JSON): 1 day
- Calibration: 1-2 days (fixture-driven)
- Refactor advice library: 1 day for initial 12-15 smells

Total: ~5-6 days. This is the v2.0 capstone — substantial work,
but the precursors (05, 06, 07) make it tractable.

## Open questions

- **`slop diagnose` vs `slop lint`.** Should diagnose REPLACE
  lint as the default? Probably not for v2.0 — lint is a known
  CI integration surface. Diagnose is the new mode; lint stays
  for backward compatibility.
- **Catalog versioning.** As fingerprints get tuned, the same
  code could be diagnosed differently across slop versions.
  Document this. Maybe ship a `--catalog v2.0.0` flag for
  reproducibility.
- **Cross-language fingerprints.** Initial catalog is Python.
  Some smells are language-agnostic (Long Method, Large Class);
  some are not (Primitive Obsession is mostly typed-language;
  Refused Bequest needs OO). Document per-language coverage in
  the catalog.

## What v2.0 looks like with this landed

- 9 metric rules (lexical), unchanged from v1.2.0
- Profile schema for cross-rule analysis
- 12-15 smell fingerprints as labelled corpus
- Matching algorithm + confidence scoring
- `slop diagnose` mode with named refactor recommendations
- Backward-compatible `slop lint` for raw rule output

The shift from v1.x to v2.0 is the shift from "metric tool" to
"diagnostic tool." Slop stops asking the user to interpret rule
output; it interprets the rules' collective output and recommends
named action.

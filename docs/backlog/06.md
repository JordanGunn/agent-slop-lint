# 06 — Smell-fingerprint fixtures: controlled measurement

**Status:** horizon (v2.0+ research) — highest single-piece payoff
**Predecessor:** [05.md](05.md) (lexicon hygiene), v1.2.0 multi-signal kernels
**Successor:** [07.md](07.md) (profile schema), [08.md](08.md) (matching)

## Context

v1.x slop produces metric data points. Running it on a real
codebase gives output of the form "rule X fired with these tokens
in these places." What it doesn't say is "this is Feature Envy"
or "this is Primitive Obsession."

The reason: slop's been an *observational* tool. We measure what's
in front of us and infer meaning from rule names + thresholds. To
make the leap to *diagnostic* output ("this looks like X smell"),
we need labelled samples — code we already know exemplifies a
specific named smell.

**Refactoring.guru** (and the broader catalog literature: Fowler's
*Refactoring* 2nd ed., Lanza & Marinescu, Mäntylä) catalogs named
smells with smelly + fixed example pairs. Building fixtures from
that corpus gives us:

- A **labelled training set**. Each fixture is "smell X."
- A **fingerprint per smell**. Run all 9 lexical rules; record
  the output as a profile vector; that's the fingerprint.
- A **baseline**. The "fixed" version of each fixture should
  produce a quiet profile. The transition from smelly to fixed
  shows the metric movement that *resolution* looks like.
- An **inverse loop**. Real codebases produce profiles; matching
  those profiles back to fingerprints turns slop's output into
  named recommendations.

This is the move from observational to **controlled** measurement.

## Scope

### Smells to include

Initial catalog (12-15 smells, prioritised by detectability via
slop's current rules):

| Smell | Source | Expected fingerprint hypothesis |
|---|---|---|
| **Long Method** | Fowler | high `verbosity`, high `stutter` (function-level), possibly high body Halstead |
| **Large Class** | Fowler | `confusion` fires (multi-receiver in one file), high WMC |
| **Feature Envy** | Fowler | `imposters` strong; cluster receiver-call density not on `self` but on a foreign type |
| **Primitive Obsession** | Fowler | `sprawl` finds an alphabet of types being passed/returned; many `tautology` hits |
| **Data Class** | Fowler | low `imposters` activity (no methods cluster on `self`); high property/getter count |
| **Refused Bequest** | Fowler | structural — DIT high but child's method-set ⊊ parent's |
| **Shotgun Surgery** | Fowler | requires churn data — `structural.hotspots` as fingerprint anchor |
| **Divergent Change** | Fowler | `confusion` (file with multi-receiver clusters) |
| **Switch Statements** | Fowler | many parallel `case`/`if` branches; `sprawl` over the switch values |
| **Lazy Class** | Fowler | small file with single trivial method; structural-ish |
| **Speculative Generality** | Fowler | high `verbosity` (long abstract names), few callers |
| **Inappropriate Intimacy** | Fowler | cross-cluster receiver-call density on internals |
| **Comments-as-smell** | Fowler | `information.section_comments` fires |
| **Long Parameter List** | Fowler | many params; could correlate with `imposters` or with absence of receiver |
| **Naming-by-Convention failure** | slop-specific | `cowards`, `hammers`, `tautology`, `slackers` all firing on the same scope |

This list mixes named smells with hypothesised fingerprints. The
work of building fixtures IS the work of testing those hypotheses.

### Fixture structure

```
docs/research/smell_fixtures/
├── README.md                — catalog + hypothesis table
├── long_method/
│   ├── smelly.py            — modified-from-textbook example
│   ├── fixed.py             — refactored target
│   └── fingerprint.md       — expected/observed profile
├── feature_envy/
│   ├── smelly.py
│   ├── fixed.py
│   └── fingerprint.md
└── ... (one directory per smell)
```

The smelly examples should be:

- **Modified from textbook examples** — refactoring.guru's are
  often deliberately contrived (single-purpose 3-line classes
  that exhibit one smell). For slop's purposes we want fixtures
  that look like real agentic code: longer, with realistic
  identifier names, not toy domains.
- **Single-smell where possible** — each fixture exhibits one
  smell cleanly so the fingerprint is interpretable. Composite
  smells get their own directory once the singletons are
  characterised.
- **Self-contained** — runnable as standalone Python; no imports
  from slop or pytest. They're textual fixtures; slop's CLI is
  pointed at them.

### Fingerprint format

Per fixture, run the full lexical rule set and record:

```
# fingerprint.md

## Smelly profile (from smelly.py)

| Rule | Fires? | Detail |
|---|---|---|
| lexical.stutter | yes | 4 violations; 3 at function scope |
| lexical.verbosity | no | — |
| lexical.cowards | no | — |
| lexical.hammers | yes | "Manager" suffix on Customer wrapper |
| lexical.tautology | no | — |
| lexical.sprawl | no | — |
| lexical.imposters | YES (strong) | profile: missing_class, body Jaccard 0.18, receiver-calls 4.2/member |
| lexical.slackers | no | — |
| lexical.confusion | no | single-cluster file |

## Fixed profile (from fixed.py)

| Rule | Fires? |
|---|---|
| lexical.stutter | no |
| ...                  | all clean (or substantially reduced) |

## Diff: what resolution looks like

- imposters cluster size: 5 → 0 (functions became methods)
- stutter: 4 → 0 (locals renamed once parent class assumes the
  context)
```

This becomes the documented expected behaviour for that smell.

## How this changes how slop works

This isn't just a new test corpus. It's a **rotation of the
analytical frame**:

- Today: rule-centric. "Which rules fire on this file?"
- Phase 6 onward: smell-centric. "Which smell does this file
  resemble?"

For a single rule, the question is binary (does it fire). For a
smell, the question becomes "how much does this file's profile
resemble the fingerprint?" — a matching problem (Phase 8).

## Relationship to existing work

- The PoC battery (`scripts/research/composition_poc_v2/`) was
  the first pass at building "what does method X find?" The
  fingerprint work is the inverse: "what does smell X look like
  to all our methods at once?"
- The cross-method evaluation in
  `docs/observations/composition/03.md` already noted that "the
  composition rules don't compete; they triangulate." Fingerprints
  formalise the triangulation.
- The profile schema (Phase 7 / [07.md](07.md)) is the data
  structure these fingerprints populate.
- The matching algorithm (Phase 8 / [08.md](08.md)) is the
  consumer.

## Implementation order

1. **Catalog draft.** List the 12-15 smells with hypothesised
   fingerprints (the table above, expanded). Half day.
2. **Fixture construction.** Build smelly + fixed pairs for the
   first 5-6 (Long Method, Feature Envy, Primitive Obsession,
   Data Class, Large Class, Refused Bequest). 1-2 days.
3. **Hypothesis testing.** Run slop against each fixture; record
   actual fingerprint; compare to hypothesised. Where they
   diverge, ask: is the rule mis-calibrated, or is the
   fingerprint hypothesis wrong? Document either way.
4. **Iteration.** Fixtures + rules calibrate against each other.
   This is the part where we learn whether slop's rules actually
   instrument the smells the literature names.
5. **Catalog expansion.** Once 5-6 fixtures stabilise, fill in
   the rest. Stragglers / partial-fingerprint smells get noted.

## Open questions

- **Granularity of "fixed."** Some smells admit multiple resolutions
  (Feature Envy can be fixed by Move Method OR by Extract Class).
  Do we record one canonical fix per fixture, or a "fixed family"?
  Probably one canonical, with notes on alternatives.
- **Cross-language fixtures.** Initial corpus is Python.
  Eventually the fingerprints should be language-agnostic
  (validated against equivalent JS/TS/Java/Ruby fixtures). Defer
  to v2.x.
- **Smell composition.** Real code often exhibits multiple smells
  on the same scope. Composite fingerprints might be
  multiplicative (smell A profile + smell B profile) or
  non-trivially combined. Keep singleton fixtures; evaluate
  composition empirically once we have a few.
- **Negative examples.** Fixtures should also include "looks
  smelly but isn't" cases — code that triggers rules for
  legitimate reasons (test fixtures, framework conventions).
  These calibrate the matching threshold for "this looks like
  X."

# 05 — Lexicon hygiene: stop-words by purpose

**Status:** horizon (v2.0+ research)
**Predecessor:** v1.2.0 multi-signal kernels
**Successor:** [06.md](06.md) (smell fingerprints), [08.md](08.md) (profile matching)

## Context

Several v1.2.0 rules consume identifier tokens for similarity or
vocabulary analysis: `lexical.imposters` computes modal-token
overlap across cluster member names; `lexical.sprawl` detects
closed-alphabet patterns; `lexical.confusion` aggregates per-file
cluster shapes. The PoC v2.5 (LDA topics) maintains a small
hand-curated stop-word list (`self`, `cls`, `args`, `kwargs`,
`name`, `value`, `data`, `true`, `false`, `none`, plus prepositions
and copulas).

Inspecting the dogfood, two stop-word concerns are conflated:

1. **Pure stop-words** — `the`, `a`, `is`, `for`, `in`. Filtered
   inconsistently. Trivial to standardise.
2. **Action-verb scaffolding** — `get`, `set`, `update`, `fetch`,
   `load`, `save`, `parse`, `validate`, `process`, `initialize`,
   `compute`, `calculate`, `create`, `build`, `make`, `render`,
   `format`. Common identifier prefixes/verbs that dominate the
   token frequency distribution.

These two concerns serve different rules in different ways. A
single stop-word list can't handle both correctly.

## The split

### Stop-words for naming-detection rules

These rules check whether a name *contains* a flagged term — the
verb is part of the signal, not noise:

- `lexical.verbosity` — flags `check_required_binaries` partly
  because of the verb `check` accumulating tokens
- `lexical.cowards` — flags `attempt_2`, `parse_old`; the verb
  is the stem we measure against
- `lexical.hammers` — banlist match; verbs aren't on the banlist,
  but content words are
- `lexical.tautology` — type-name suffix match

For these, the **minimal** stop-word set (current state) is
correct. Don't filter verbs.

### Stop-words for profile-analysis rules

These rules compare distributions of tokens to detect cluster
shape — verb prevalence is noise that swamps the signal:

- `lexical.imposters` modal-token analysis (which tokens dominate
  cluster member names?)
- `lexical.sprawl` alphabet detection (when many `get_*`,
  `set_*`, `fetch_*` patterns exist, the alphabet is the *suffix*
  of those verbs, not the verb itself)
- LDA topic discovery
- (Future) profile fingerprints for smell matching

For these, an **aggressive** stop-word set that strips action
verbs gives cleaner profiles.

## Proposed structure

```
src/cli/slop/_lexical/_stopwords.py
```

```python
NAMING_DETECTION_STOPWORDS: frozenset[str] = frozenset({
    "the", "a", "an", "is", "are", "to", "for", "in", "on", "of",
    "with", "by", "as", "at", "and", "or", "but", "if",
    # No verbs — verbs are signal for naming-discipline rules.
})

PROFILE_ANALYSIS_STOPWORDS: frozenset[str] = (
    NAMING_DETECTION_STOPWORDS | frozenset({
        # Action verbs and identifier scaffolding
        "get", "gets", "set", "sets", "update", "updates",
        "fetch", "fetches", "load", "loads", "save", "saves",
        "store", "stores", "parse", "parses", "validate",
        "validates", "process", "processes", "initialize",
        "init", "compute", "computes", "calculate", "calculates",
        "create", "creates", "build", "builds", "make", "makes",
        "render", "renders", "format", "formats", "extract",
        "extracts", "collect", "collects", "find", "finds",
        "check", "checks", "ensure", "ensures",
        # Generic role nouns that appear constantly
        "self", "cls", "args", "kwargs", "value", "data",
        "result", "results", "item", "items",
        # Frequently-used helpers
        "func", "fn", "method", "callable", "callback",
        "handler", "wrapper",
    })
)
```

(Both lists curated from observation, not hard-bounded — expected
to evolve.)

## Audit of consumers

| Site | Current stop-word handling | Should use |
|---|---|---|
| `_lexical/imposters.py` modal-token analysis | none | `PROFILE_ANALYSIS_STOPWORDS` |
| `_lexical/sprawl.py` alphabet detection | none | `PROFILE_ANALYSIS_STOPWORDS` (verbs in stem positions are noise) |
| `_lexical/hammers.py` banlist match | none (correct) | none — verbs aren't being matched |
| `_lexical/cowards.py` suffix match | none (correct) | none — suffixes are the point |
| `_lexical/tautology.py` type match | none (correct) | none — types aren't stop-words |
| `_lexical/stutter.py` token overlap | implicit (`_` and digit boundaries only) | `NAMING_DETECTION_STOPWORDS` (subtle: `get_user` repeating with `get_user_email` SHOULD flag, but `get` shared with `get_X` shouldn't dominate) |
| `_lexical/verbosity.py` token count | none (correct) | none — counting verbs is part of the metric |
| PoC v2.5 LDA | minimal hand-curated list | `PROFILE_ANALYSIS_STOPWORDS` |
| PoC v2.1 dominant text | minimal | `PROFILE_ANALYSIS_STOPWORDS` |

## Implementation steps

1. Create `_lexical/_stopwords.py` with the two named sets.
2. Import + apply in the consumers that should use the aggressive
   set (imposters, sprawl, future profile work).
3. Add a `stop_word_profile: str = "naming"` parameter to the
   PoC scripts and rule wrappers where the choice is non-obvious;
   default per their natural use.
4. Update v2 PoC outputs (`scripts/research/composition_poc_v2/output_*.md`)
   by re-running with the new stop-words; record diffs in
   observation 04 (or 05).
5. Document in `docs/methods/lexical/` how stop-words affect each
   method.

## Open questions

- **Calibration source.** The aggressive stop-word list is
  empirical. Do we calibrate against:
  - frequency distribution on a single corpus (slop dogfood)
  - frequency distribution across the calibration corpora (Phase 6
    of v1.2 plan, deferred)
  - cross-language analysis (some "scaffolding" verbs differ:
    `fetch` is more Ruby-ish, `dispatch` is more JS-ish)
- **Configurability.** Should users be able to override the
  stop-word lists per-project? Probably yes — domain-specific
  vocabularies (a banking codebase's `withdraw`, `deposit`,
  `transfer` are domain verbs, not scaffolding).
- **Stemming.** `gets` and `get` should both filter. Do we
  stem before stop-word check, or list both forms? Listing both
  is more precise but verbose; stemming is cheap but introduces
  another knob.

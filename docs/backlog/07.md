# 07 — Profile schema: the unit of v2.0 analysis

**Status:** horizon (v2.0+ research)
**Predecessor:** v1.2.0 multi-signal kernels, [05.md](05.md), [06.md](06.md)
**Successor:** [08.md](08.md) (matching + diagnostic output)

## Context

Today each rule produces violations independently. To answer
"what does the whole rule set say about this file?" you have to
join violations across rules manually. Cross-rule correlation
findings — like "the `lexical.verbosity` cluster IS the same
cluster `lexical.imposters` flagged" — aren't surfaced; the user
has to notice on their own.

The Phase 6 fixture work needs a structural answer to this. To
fingerprint a smell, we need to compare *the whole profile of
rule outputs* across smells — not violation lists per rule, but
a single record per scope summarising what every rule says.

## Definition

A **profile** is a structured snapshot of rule outputs at a given
scope. Three scope levels matter:

- **Cluster profile** — for one parameter cluster (per
  imposters / sprawl / slackers): what rules fire on this cluster,
  with what numeric values?
- **File profile** — for one source file: aggregate of
  cluster profiles + per-file rules (`confusion`, file-level
  `verbosity` aggregates).
- **Codebase profile** — for the whole corpus: aggregate file
  profiles + global signals (alphabet recurrence à la PoC v2.3).

## Schema sketch

```python
@dataclass
class ProfileSlice:
    """One rule's contribution to a scope's profile."""
    rule: str                    # e.g., "lexical.imposters"
    fired: bool                  # any violations at this scope?
    severity: str | None         # severity if fired
    metrics: dict[str, float]    # rule-specific numbers
    findings: list[str]          # human-readable summaries (truncated)


@dataclass
class ScopeProfile:
    """All rules' outputs at one scope."""
    scope: str                   # e.g., "cli/slop/output.py"
    scope_kind: str              # "cluster" | "file" | "package" | "root"
    slices: dict[str, ProfileSlice]   # rule name → slice
    cross_refs: list[tuple[str, str]] # ("rule_a", "rule_b") pairs
                                      # pointing at the same artifact
```

The `metrics` dict per rule is the key: instead of just "fired or
not," each rule reports its numeric output. For
`lexical.imposters` that's body-Jaccard, receiver-call density,
modal-overlap. For `lexical.verbosity` it's the token count. For
`lexical.sprawl` it's the alphabet size and concept density.

The `cross_refs` list is what enables observation 03's
triangulation insight: when the same file/cluster fires multiple
rules, the profile records that the rules are looking at the
*same artifact* — not coincidentally co-firing. This is what
turns 9 violation lists into "here's what this artifact looks
like through every probe at once."

## What this enables

### 1. Fingerprint comparison ([06.md](06.md))

Each smell fixture's profile is a `ScopeProfile`. Comparing two
profiles becomes a structured operation: align by rule, compare
metric vectors, compute distance.

### 2. Rule correlation reporting

A new output mode aggregates by scope across rules:

```
output.py
  ├── verbosity: 4 violations (max tokens 5)
  ├── imposters: 3 clusters (1 missing_class, 2 heterogeneous)
  ├── slackers: 2 clusters (no naming template)
  └── stutter: 6 violations (5 at function scope)
  CROSS-REFS:
    - imposters[result] = verbosity[format_human, format_quiet]
                        = stutter[format_human → output module]
```

Reading the file's profile collectively, instead of nine
independent rule reports, surfaces what each rule says about the
same artifact.

### 3. Severity composition

A scope where 5 rules fire mildly might warrant the same review
attention as one where 1 rule fires hard. Profile aggregation can
compose severities — "low across many" vs "high on one" become
distinguishable signals.

### 4. Diagnostic output ([08.md](08.md))

The matching algorithm consumes profiles, not raw rule outputs.
Profile is the lingua franca.

## Producer changes

Each rule currently returns a `RuleResult` with violations +
summary. To populate profiles cleanly:

1. Add a per-rule method `to_profile_slice(scope)` that produces
   the profile slice for a given scope. Most rules already have
   the data internally; this is a structured export.
2. The profile aggregator runs all rules, then walks each scope
   and assembles a `ScopeProfile` from the slices.
3. Cross-refs are computed by checking whether multiple rules'
   findings share a `(file, line)` or `(file, symbol)` tuple.

## Consumer surface

```python
from slop.profile import collect_profiles

profiles = collect_profiles(root, slop_config)
# profiles: dict[scope_str, ScopeProfile]

for scope, profile in profiles.items():
    if profile.fingerprint_distance(SMELL_FEATURE_ENVY) < 0.3:
        print(f"{scope} resembles Feature Envy")
```

The `fingerprint_distance` function lives in Phase 8 / 08.md.

## Backward compatibility

Profile work is additive — existing `RuleResult` API and CLI
output unchanged. Profile output is a new mode (`slop diagnose`
or `slop lint --output profile`) that consumers opt into.

## Implementation order

1. Define `ScopeProfile` and `ProfileSlice` dataclasses.
2. Add `to_profile_slice` methods on 2-3 rules as proof-of-concept.
3. Profile collector that walks the registry and aggregates.
4. JSON output mode for profiles (consumed by Phase 6 fixtures
   to record fingerprints, and by Phase 8 matching).
5. Migrate remaining rules to populate profile slices.

## Open questions

- **Cluster identity.** Profile keys depend on stable cluster
  identifiers. Today clusters are tagged by `(scope, parameter_name)`
  for imposters; sprawl uses `(scope, alphabet)`. Need a unified
  identity scheme so cross-rule correlation can match.
- **Metric normalisation.** Rule metrics are heterogeneous —
  body-Jaccard is in [0, 1], token count is unbounded. Distance
  computation requires either per-metric normalisation or
  rule-specific weighting. Both have empirical implications.
- **Aggregation across scopes.** A file-scope profile aggregates
  cluster-scope profiles for clusters in that file; a package-
  scope profile aggregates files. The aggregation operator
  isn't obvious — sum, max, mean? Likely depends on the metric.

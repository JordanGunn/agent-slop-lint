# 03 — Compressed output for high-volume rules

**Status:** backlog (not scheduled)
**Target:** v1.2.x

## Context

The v1.1.0 dogfood (recorded in
[`docs/observations/dogfood/01.md`](../observations/dogfood/01.md))
exposed a real ceiling on the human formatter: when a rule produces
~100s of advisories on a single codebase, the per-violation listing
stops being actionable.

`lexical.name_verbosity` produced 380 advisories on slop's own
602-function corpus. The user sees the first five and "...and 375
more." There is no representation in the current formatter that
expresses what the 380 actually are — which is a small number of
architectural debts (a `LegacyConfigMigrator` candidate, a
`RuleResultRenderer` candidate, an AST-walker family, and a few
others) producing many surface symptoms.

Until this is addressed, high-volume rules will tend to be waived
rather than acted on. The measurement is honest, but the report
cannot drive the work it points at.

The correct response is **not** to inflate thresholds (the volume
itself is part of the signal — see
`feedback_thresholds_are_honest.md`). The response is to let the
formatter compress.

## What "actionable" looks like

For high-volume rules, individual hits are not the unit of action.
The unit is one of:

- **Source-location group** — "31 of these 380 hits live in
  `_compat.py`. Showing the worst three; rest collapsed."
- **Architectural cluster** — "These 6 functions all take
  `raw_rules: dict` as their first parameter. Same cluster as
  reported by `composition.first_parameter_drift`."
- **Token group** — "47 of these names contain `legacy_table` or
  `legacy_rule` as a substring. Likely candidate namespace:
  `Legacy`."
- **Threshold-band** — "32 names are 4 tokens, 9 are 5 tokens,
  2 are 6 tokens. Worst offenders shown."

The right grouping depends on the rule. The general shape is
"summary header + worst N exemplars + tail count," with the
header doing the storytelling.

## Proposed approach

### Per-rule compression metadata

Add an optional `compression` field to `RuleResult`:

```python
@dataclass
class CompressionGroup:
    label: str             # "_compat.py", "raw_rules cluster", "4-token names"
    exemplars: list[Violation]   # worst N hits to display
    suppressed_count: int        # how many more in this group
    metadata: dict[str, Any]     # rule-specific context

@dataclass
class RuleResult:
    ...
    compression: list[CompressionGroup] | None = None
```

When `compression` is populated, the human formatter renders the
groups instead of the flat violation list. Rules without a
compression strategy fall through to current behaviour. Migration
is per-rule and incremental.

### Rule-specific groupers

Each high-volume rule decides its own grouping. Implementations live
alongside the rule wrapper:

- `lexical.name_verbosity`: group by source file, sorted by token
  count desc. Header counts hits per file.
- `lexical.stutter.identifiers`: group by enclosing function family
  (functions whose body identifiers all stutter together).
- `lexical.type_tag_suffixes`: group by the matched type
  (`_path → Path`, `_dict → dict`, etc).
- `composition.first_parameter_drift`: already grouped by parameter
  name in the kernel; current rendering is fine.
- `lexical.numbered_variants`: group by suffix kind (`_old`/`_new`
  pairs together; `_v1`/`_v2` together).

### Cross-rule correlation (optional, follow-up)

The dogfood observation noted that `lexical.name_verbosity` and
`composition.first_parameter_drift` cross-corroborate — the
`_compat.py` long-name cluster IS the `raw_rules` first-parameter
cluster. A future formatter mode could detect when groups from
different rules overlap on file/function/symbol and surface that
("These 6 long names are the same functions
`composition.first_parameter_drift` flagged as a hidden class.").

This is harder and out of scope for the initial compression work.
Worth recording as a direction.

### CLI ergonomics

A new flag `--max-violations 0` already disables truncation; the
compression mode would be a separate axis:

```bash
slop lint --output human                # default: per-violation
slop lint --output human --compress     # grouped
slop lint --output human --compress=files   # specific grouping
slop lint --output json                 # full data; compression metadata
                                        # included if rule emits it
```

JSON mode always emits the full violation list. Compression is a
human-formatter feature.

## Why this matters

The user's framing: "as groups are addressed, others become more
obvious." A compressed renderer naturally surfaces the next
architectural target as the current one is fixed. A flat per-
violation listing cannot — every individual hit looks the same
shape; the reader has no way to see which one represents 50 sibling
hits and which represents 1.

This is also a defensive feature against the "tool generates noise,
we waive it" failure mode. A compression mode that says "31 hits
in `_compat.py`, here are the three worst" makes the action
obvious. A flat list of 380 hits invites the project-wide waiver.

The v1.1.0 release dogfood's `lexical.name_verbosity` waiver is
exactly this failure mode in miniature — the right response was
"this volume is real; here's the cluster" but the formatter had no
way to say that. So the response was a global waiver with prose
explaining what the report should have said directly.

## Out of scope (for the initial compression work)

- Reorganising the underlying kernels to produce groups instead of
  individual violations. Compression is a presentation concern;
  rule kernels keep producing per-occurrence data.
- Cross-rule correlation (deferred to later; see above).
- A separate `--output groups` format that bypasses violations
  entirely. Extra formatter state machine; postpone.
- Backwards-compat concerns. Adding optional `compression` field is
  additive; older consumers ignore it.

## Implementation order

1. Extend `RuleResult` with optional `compression` field.
2. Update `output.format_human` to detect populated compression and
   render groups. Fall through to current per-violation rendering
   otherwise.
3. Implement compression on `lexical.name_verbosity` first (highest-
   volume rule; clearest grouping by file). Validate on the slop
   dogfood: does the output drive the user toward the
   `_compat.py` cluster?
4. Apply to the other high-volume rules
   (`lexical.stutter.identifiers`, `lexical.stutter.namespaces`,
   `lexical.type_tag_suffixes` for codebases without pytest noise).
5. Add CLI flag.
6. Document in `docs/CONFIG.md` and rule pages.

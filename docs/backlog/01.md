# 01 — Lexical rule overhaul: split stutter, add agent-tell rules

**Status:** backlog (not scheduled)
**Target:** v1.1.0 (or a dedicated lexical release)

## Context

A v1.0.3 dogfood pass surfaced two structural critiques of the
existing lexical rules:

- `lexical.stutter` generated 158 advisories on slop's own corpus and
  required 5 waivers in `.slop.toml` to keep the noise tolerable.
  The waiver volume is itself a signal: the rule fires on patterns
  that are real smells (agents over-naming local variables) but the
  user has been writing waivers because the per-instance impact is
  low and there was no way to dial that variant down.
- `lexical.tersity` produced zero violations and is a preemptive
  guard, not actively useful. Keep as-is — agents over-correcting
  toward terse names is rare and usually justified
  (`for i in range(10)` should not be flagged).
- `lexical.verbosity` uses *mean* tokens per identifier; a single
  long horror like `compute_total_aggregated_user_score_value`
  averages out under several normal names. Median/max would catch
  it, but the deeper finding is that the function-name length is
  itself the most visible signal and deserves its own rule.

A long function name is usually a class-without-class:
`check_required_binaries` is three tokens because the namespace it
should belong to doesn't exist yet. If `Preflight` were a class, the
same logic becomes `Preflight.check()`.

The waivers themselves correctly identified pragmatic exceptions
that **shouldn't be exceptions** — the underlying patterns are noise.
The right fix is to split the rule and let the user dial down the
variant they don't care about, not to keep accumulating waivers.

## Items (in rough order of value)

### 1. Split `lexical.stutter` into three rules

Drop the single rule. Replace with:

- **`lexical.stutter.namespaces`** — symbol name stutters with module
  path. Examples: `slop.rules.complexity.complexity_kernel`,
  `_lexical.identifier_tokens.identifier_token_kernel`. Highest
  signal-to-noise; the module already names the thing.
- **`lexical.stutter.callers`** — method or attribute name stutters
  with its enclosing class. Examples: `UserService.get_user_user_id`,
  `Animal.animal_name`. Strongest agent tell of the three.
- **`lexical.stutter.identifiers`** — local variable name stutters
  with the enclosing function. Examples: `legacy_table` inside
  `_apply_one_legacy_table`, `required_binaries` inside
  `check_required_binaries`. Lower per-instance impact but high
  frequency. User-disableable; default `severity = "warning"` (or
  `"info"` for teams that treat it as a soft signal).

Migration: drop the 5 stutter waivers from `.slop.toml`. The
patterns they cover all fall under `stutter.identifiers`, which can
be configured per-project.

### 2. `lexical.type_tag_suffixes`

Flag identifiers whose suffix re-states the type annotation. The
type system already says it; the suffix is ornamentation.

Examples:
- `result_dict: dict[str, int]` — suffix `_dict` matches the
  annotation's outer type.
- `config_path: Path` — suffix `_path`.
- `params_list: list[Param]` — suffix `_list`.
- `user_obj: User` — suffix `_obj`.
- `data_str: str` — suffix `_str`.

Detection: parse the type annotation, normalize to its outer
constructor (`dict`, `list`, `str`, `Path`, etc.), check whether the
identifier's last underscore-separated token matches a normalized
form of that type.

False positives to handle: identifiers where the suffix is part of a
real domain term (`username` ≠ `user_name` ≠ `user_str`). Restrict
flagging to identifiers ending in `_dict`, `_list`, `_str`, `_path`,
`_obj`, `_data`, `_set`, `_tuple`, `_int`, `_float`, `_bool` after
underscore parsing.

### 3. `lexical.name_verbosity`

Flag functions or classes whose **own** name has more than N tokens
(default 3). Independent from `lexical.verbosity` (which measures
body identifiers).

Rationale: a function called `check_required_binaries_for_python_runtime`
is doing class-level work without a class. The deep reading is that
the namespace it should live in doesn't exist; the function name
re-creates the namespace inline.

Heuristic: 3 tokens is the edge (`check_required_binaries` already
borderline); 4+ tokens is almost always a misplacement.

Detection: tokenize the function/class name (snake_case + CamelCase
split, same as `lexical.verbosity`), count tokens. Default threshold
3 with `severity = "warning"`.

### 4. `lexical.boilerplate_docstrings`

Flag docstrings that restate the function name with no new
information.

Examples:
- `def get_user_email():\n    """Get the user email."""`
- `def parse_config():\n    """Parses the config."""`
- `def initialize_logger():\n    """Initialize the logger."""`

Detection: tokenize the function name and the docstring's first
sentence. If the docstring's content tokens (excluding articles,
prepositions, "the/a/an") are a subset of the function name's
tokens, flag.

Strong agent tell — a habit of "every function gets a docstring"
without considering whether the docstring adds anything.

### 5. `lexical.numbered_variants`

Flag identifiers ending in numeric or alphabetic disambiguators that
suggest an agent named two related things by sequencing them rather
than describing them.

Examples:
- `result1`, `result2`
- `_v1`, `_v2`
- `attempt_1`, `attempt_2`
- `solve_a`, `solve_b`
- `_old`, `_new`
- `_local` (often a copy-paste rename to dodge a name collision —
  e.g., `_cpp_fn_name_local` exists because `_cpp_fn_name` was taken
  in another module)
- `_inner`, `_alt`, `_helper` when used as disambiguation suffixes

Detection: regex on identifiers ending in `_?\d+`, `_v\d+`,
`_attempt\d+`, plus a configurable list of disambiguation suffixes
(`_old`, `_new`, `_local`, `_inner`, `_alt`, `_helper`, `_temp`).

### 6. `lexical.identifier_singletons`

Flag functions where most named locals are used exactly once on a
subsequent right-hand side.

Example to flag:

```python
def process_user_request(req):
    parsed_request = parse(req)
    user_id = parsed_request.id
    user_record = lookup(user_id)
    audit_log_entry = build_log_entry(user_record)
    write_audit(audit_log_entry)
    response_payload = build_response(user_record)
    return response_payload
```

Six distinct locals; four (`parsed_request`, `user_id`,
`audit_log_entry`, `response_payload`) are each used exactly once
after assignment. Reading this requires holding six fresh names in
your head; none accumulate meaning through reference.

The same logic without singleton bindings:

```python
def process_user_request(req):
    user = lookup(parse(req).id)
    write_audit(build_log_entry(user))
    return build_response(user)
```

`user` carries through; verbs do the work.

Metric: fraction of named locals whose subsequent right-hand-side
references count is exactly 1, divided by count of distinct named
locals. Default threshold 0.6 with `severity = "info"`.

Cases NOT to fire:
- The singleton local IS the return value (`x = compute(); return x`
  is fine; `return compute()` is a stylistic choice, not slop's
  call).
- Pipeline functions where each named intermediate is genuinely
  complex enough that extraction aids debugging (filterable: if the
  local is the only argument to a call AND the call is multi-line,
  exempt).

Implementation cost: higher than the others — requires use-count
tracking across the function body, not just identifier scanning.

## Open questions

- Severity defaults: `lexical.stutter.identifiers` and
  `lexical.identifier_singletons` should probably default to
  `severity = "info"` rather than `"warning"`, since they're
  high-frequency soft signals. The other four are clearer
  warnings.
- Backwards compatibility: `lexical.stutter` becoming three rules is
  a breaking change for anyone with `[rules.lexical.stutter]` in
  their config. The compat shim from v0.9.0 → v1.0.0 already
  established a translation pattern; we'd add `lexical.stutter` →
  `lexical.stutter.callers` (the closest single-rule match) at the
  config-load layer.

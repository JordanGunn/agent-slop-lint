# 02 — `lexical.weasel_words`: configurable banlist for catchall identifier terms

**Status:** backlog (not scheduled)
**Target:** v1.1.0 alongside `01.md` lexical work
**Suite:** `lexical`

## Context

Catchall terms — `Manager`, `Helper`, `Utility`, `Spec`, `Handler`,
`Service`, `Wrapper`, `Object`, `Item`, `Data`, etc — are
single-word agent tells. They show up when the agent needed a noun
and didn't have one. The harm is two-stage:

1. The original use is a missed naming opportunity.
2. The presence of `FooManager` makes it natural to dump every
   subsequent unrelated responsibility into `FooManager`. The catchall
   becomes a gravitational sink for future responsibilities that
   don't have a clear home.

The textbook self-example: while drafting `01.md`, the term
``LanguageSpec`` was used to describe the proposed abstraction. The
`Spec` suffix was reflexive — `Language` was sufficient on its own
(no shadowing risk in a linter context). That single keystroke is
the entire failure mode this rule targets.

## Rule design

### Detection

Per-identifier check: tokenize identifier (snake_case + CamelCase
split). For each token, check against the configured banlist. If a
matching term appears at a position covered by the rule's position
config, flag.

### Configuration shape

Per-word config, not a flat list. Each entry specifies:

- `word` — the term to match (case-insensitive against tokens).
- `positions` — list of `"prefix" | "suffix" | "any" | "module_name"`.
  - `"prefix"` — flags when the term is the first token (e.g.,
    `HelperFoo`).
  - `"suffix"` — flags when the term is the last token (e.g.,
    `FooHelper`, `UserManager`).
  - `"any"` — flags any position including standalone whole-word
    use.
  - `"module_name"` — flags when the term is the file/module name
    itself (`utils.py`, `helpers.py`, `common.py`, `core.py`).
- `severity` — per-word override of the rule's default severity.
- `exempt_when` — list of context predicates that suppress the
  flag. Initial set: `"module_is_test"`, `"module_is_main"`. (Tests
  legitimately use `*Spec` naming for behaviour-driven suites.)

### Sample TOML

```toml
[rules.lexical.weasel_words]
enabled = true
severity = "warning"  # default for any term without per-word override

[[rules.lexical.weasel_words.terms]]
word = "Manager"
positions = ["suffix"]
severity = "warning"

[[rules.lexical.weasel_words.terms]]
word = "Util"
positions = ["any", "module_name"]
severity = "warning"

[[rules.lexical.weasel_words.terms]]
word = "Object"
positions = ["suffix"]
severity = "error"  # zero semantic content; no excuse

[[rules.lexical.weasel_words.terms]]
word = "Spec"
positions = ["suffix"]
severity = "warning"
exempt_when = ["module_is_test"]

[[rules.lexical.weasel_words.terms]]
word = "Abstract"
positions = ["prefix"]
severity = "info"  # ABC pattern is legit
```

### Default shipped banlist

| Term | Positions | Default severity | Notes |
|---|---|---|---|
| `Manager`, `Coordinator` | suffix | warning | DI codebases use legitimately; per-team tunable |
| `Helper`, `Utility`, `Util`, `Utils` | any, module_name | warning | almost never a real noun |
| `Handler`, `Processor` | suffix | info | sometimes a legit role |
| `Service`, `Provider`, `Engine` | suffix | info | DDD-legitimate; controversial in non-DDD codebases |
| `Factory`, `Builder` | suffix | info | named patterns; tunable |
| `Wrapper`, `Adapter` | suffix | info | sometimes pattern, sometimes drift |
| `Spec`, `Specification` | suffix | warning | the canonical reflex agent term |
| `Base` | suffix | warning | suspect when not actually abstract |
| `Abstract` | prefix | info | ABC pattern is legit |
| `Object`, `Item`, `Element`, `Thing` | suffix | error | zero semantic content |
| `Data`, `Info` | suffix | warning | restates obvious context |
| `Container`, `Holder` | suffix | warning | empties of meaning |
| `Common`, `Core`, `Misc`, `Extra`, `Shared` | module_name, suffix | warning | dumping-ground module names |
| `Stuff`, `Things` | any | error | always a smell |

### Profiles

- **`default`** — the table above. Controversial DDD terms at `info`,
  textbook smells at `warning`, zero-content terms at `error`.
- **`lax`** — downgrade DDD terms (`Service`, `Provider`, `Manager`,
  `Engine`) to `info`. Suitable for Spring / Rails / Symfony /
  Django-DDD codebases.
- **`strict`** — promote `Manager`, `Helper`, `Util` to `error`;
  add `Class`, `Type`, `Kind` (when used as suffixes after a
  concrete noun) to the warning tier.

## Why "weasel words"

`lexical.weasel_words` over `lexical.catchall_terms`,
`lexical.banned_terms`, or `lexical.naming_drift`:

- **Weasel words** has decades of usage in English-prose criticism
  (technical writing, journalism). The term carries the right
  connotation: words that *seem* informative but evade meaning.
- It's specific. `catchall_terms` is descriptive but bland.
  `banned_terms` puts the focus on the mechanism, not the intent.
  `naming_drift` is too abstract — doesn't say what to look for.
- The rule's intent is immediately legible from the name.

## Position split rationale

Position matters because the same term is a legit pattern at one
position and a smell at another:

- `Helper` as a suffix (`MyHelper`) is the agent reflex: gave up on
  naming the noun.
- `Helper` as a prefix (`HelperFoo`) is uncommon; usually a smell
  the other way.
- `Helper` as a whole word may be a legitimate entity name in some
  codebases.
- `Abstract` as a prefix is the ABC pattern.
- `Abstract` as a suffix is noise (`FooAbstract` doesn't help).

A flat banlist would miss these distinctions. The per-word
position list is essential.

## Implementation notes

- **Lexical, not composition.** Composition rules detect
  cross-identifier patterns (`affix_polymorphism`,
  `first_parameter_drift`). Catchall words are about individual
  identifier vocabulary — that's lexical.
- **Tokenisation is shared with existing lexical rules.** Reuse
  `_lexical/identifier_tokens.py:split_identifier`.
- **Module-name positioning** requires file-path inspection in
  addition to identifier scanning. New code path; small cost.
- **Per-word context predicates** (`exempt_when`) are a small
  callable hook in the kernel. Initial implementations: regex on
  file path for `module_is_test` and `module_is_main`.

## Open questions

- Profile vs explicit config: should a project pick a profile
  (`default` / `lax` / `strict`) and have it expand to the full
  per-word config at load time, or always require explicit per-word
  entries? Profiles are friendlier; explicit is more honest. Lean
  toward profiles, with the option to override individual terms.
- Allowlists for legitimate single uses: a codebase may have one
  `MyOnlyManager` that's genuinely the right name. A
  per-occurrence waiver covers this (existing waiver mechanism), so
  no special config needed.
- Internationalisation: the default list is English. Codebases in
  other languages would need their own banlist. Out of scope for
  initial ship; user-extendable via the existing config schema.

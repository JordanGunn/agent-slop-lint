# 09 ŌĆö Per-language ecosystem idiom layer

**Status:** horizon (post-v1.x lexicon work)
**Predecessor:** [05.md](05.md) (lexicon hygiene), Newman 2017 + Fan 2023
**Successor:** none yet

## Context

The lexicon work establishes a layered ignore model (see
`docs/research/identifier-vocabulary.md`):

- Layer 1 ŌĆö universal noise (Newman 14 + glue words). Shipped.
- Layer 2 ŌĆö SE boilerplate (`manager`, `helper`, `service`, ...).
  Deliberately excluded ŌĆö `lexical.hammers`'s job.
- Layer 4 ŌĆö **ecosystem idioms.** Per-language scaffolding tokens
  that are high-frequency *within* an ecosystem but absent or rare
  in others. Deferred.
- Layer 5 ŌĆö corpus-derived TF-IDF / Poisson. Deferred until Phase
  06 fixture corpora exist.

Layer 4 is the open work captured here.

## What an ecosystem-idiom set looks like

Sketches per ecosystem, drawn from informal observation and the
literature dive (specifically the comments in
`docs/research/identifier-vocabulary.md`'s sources):

| Ecosystem | Idiomatic identifiers |
|---|---|
| Python | `self`, `cls`, `args`, `kwargs`, `__init__`, `__main__` |
| Go | `ctx`, `err`, `req`, `resp`, `svc`, `cfg` |
| C / C++ | `ptr`, `buf`, `len`, `idx`, `num`, `str`, `impl`, `fn` |
| Java | `this`, `super`, `obj`, `tmp` |
| JS / TS | `this`, `cb`, `fn`, `el`, `evt`, `e` |
| React (TS/JS) | `props`, `state`, `ref`, `hook`, `component`, `children` |
| Ruby | `self`, `attr`, `instance` |
| Rust | `self`, `Self`, `mut`, `iter`, `ref` |

These are deliberately illustrative, not a final list. The list
should be derived empirically rather than hand-curated when we
land it (see "Calibration" below).

## Why this is its own layer (not Layer 1)

The user's original framing nailed it: *"these should not be
globally ignored. They are often high-frequency but highly
diagnostic within an ecosystem."*

`ctx` and `err` are noise *for* a Go codebase but *signal* in the
sense that finding them in a TypeScript codebase would be
remarkable. Conflating Layer 1 (cross-corpus universal) with Layer
4 (per-language idiomatic) loses that signal:

- **Cross-language sprawl detection** ŌĆö if a multi-language repo
  has parallel `extract_python`, `extract_go`, `extract_rust`
  patterns, sprawl wants the *language tokens themselves* preserved
  as the alphabet members. Layer-4 noise lives in the same files
  but is per-file, not cross-file.
- **Alien idiom detection** ŌĆö a future rule might want to flag
  "this Python codebase is using Go-style `ctx` patterns" as a
  signal of unidiomatic translation. Baking Go idioms globally
  would silence that detection.

## Implementation shape

### Option A ŌĆö per-language constants, looked up by detected language

```python
# _lexical/_words_<lang>.py ŌĆö one module per language
# _lexical/_words.py exposes a registry

LANG_IDIOMS: dict[str, frozenset[str]] = {
    "python": PYTHON_IDIOMS,
    "go": GO_IDIOMS,
    "rust": RUST_IDIOMS,
    "javascript": JS_IDIOMS,
    "typescript": JS_IDIOMS | REACT_IDIOMS,  # composition
    ...
}

def idiom_set(language: str) -> frozenset[str]:
    return LANG_IDIOMS.get(language, frozenset())
```

Rules that want per-language filtering call `idiom_set(lang)` and
union with `UNIVERSAL_NOISE` for the `exclude=` argument.

### Option B ŌĆö mixin-style composition on Lexeme

Each `Lexeme` carries its language tag. Lexicon constructors can
accept an `apply_idioms: bool = False` flag that, when true,
unions per-Lexeme language idioms into the per-query exclude set.
More complex; couples Lexicon to Lexeme-language metadata.

**Lean: Option A.** Simpler, language-set is a property of the
*query*, not of the Lexicon. Stays consistent with "filtering is a
per-rule opt-in."

## Calibration

The hand-sketched lists above are starting points, not ship-ready.
Two ways to derive them:

1. **Per-language dogfood corpora.** Same Phase 06 work that
   produces smell fixtures should produce per-language reference
   codebases. Run frequency analysis over each; identifiers
   appearing in Ōēź 80% of files across the corpus *and* having no
   structural signal under hammers/cowards/tautology are
   idiom-set candidates.
2. **Literature-grounded scaffolding-verb list** (Newman 2017
   Table I + Dragoumi stereotypes). The verb stems `get`, `set`,
   `predicate`, `property`, `command`, `factory`, `controller`
   are language-agnostic but ecosystem-flavoured (e.g. Java
   leans heavily on `getX`/`setX`; Go avoids them).

## Why not now

Three reasons:

1. **Slop's primary corpus is Python.** Until v2.0 expands the
   smell-fixture work to multi-language, per-language idiom sets
   add design surface without proportional payoff.
2. **Empirical grounding requires corpora we don't have.**
   Hand-curating Layer 4 today is the same kind of bias that
   `StopwordProfile` would have introduced ŌĆö speculative noise.
3. **Lexicon design works without it.** Rules that need
   per-language filtering can pass an explicit `exclude=` set
   today. Layer 4 is convenience, not capability.

## When to revisit

When any of these triggers:

- A rule fires noisily on a non-Python codebase due to ecosystem
  idioms (e.g. `lexical.imposters` over-firing on Go's `ctx err`
  parameter pattern).
- Phase 06 produces multi-language fixture corpora that can
  calibrate the lists.
- A new rule needs idiom-aware behaviour (e.g. detection of
  cross-ecosystem idiom transplants).

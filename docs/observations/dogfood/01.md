# Dogfood observation: v1.1.0 release

**Date:** 2026-05-06
**Branch:** `feature/v1.1.0-composition-and-lexical`
**Corpus:** slop itself (~602 Python functions across `src/cli/slop/`
and `src/tests/`).

## Context

slop was written almost entirely by AI agents, with limited human
intervention beyond direction-setting, scope guidance, and review.
This was deliberate. The thesis of the tool — that agent-written
code accumulates structural and lexical patterns that human-written
code largely avoids, and that those patterns are detectable from
identifier and AST signal — depends on the tool being able to surface
those patterns when it sees them. A linter that targets agent code
should be able to find significant signal in code an agent wrote.

The v1.1.0 release adds 8 new rules across two suites
(`composition.*` and an expanded `lexical.*`) plus the three-way
`lexical.stutter` split. Running these rules against slop's own
codebase is therefore both a dogfood and a partial test of the
thesis: if slop produces a rich set of actionable findings on
itself, the tool's framing is supported. If it produces nothing,
either the codebase is unusually disciplined or the rules are
poorly calibrated.

## Findings (post-config, pre-individual-waiver)

Total functions analysed: 602 (post `.slop.toml` exclusions of
kernel subpackages).

| Rule | Hits | Notes |
|---|---|---|
| `composition.first_parameter_drift` | 13 strong | 11 in source, 2 in tests (waived as pytest fixtures) |
| `composition.affix_polymorphism` | 2 concepts | False positives from test/source naming overlap |
| `lexical.name_verbosity` | 380 | 4+ token function names, all waived in dogfood |
| `lexical.stutter.identifiers` | 212 | Local-variable stutter with enclosing function name |
| `lexical.stutter.namespaces` | 124 | Symbol-name stutter with module path |
| `lexical.stutter.callers` | 0 | Slop has few class-scope methods |
| `lexical.boilerplate_docstrings` | 4 | Docstrings that just restate the function name |
| `lexical.identifier_singletons` | 4 | Functions where most locals are write-once-read-once |
| `lexical.type_tag_suffixes` | 350 | All waived — `tmp_path: Path` is a pytest fixture name |
| `lexical.numbered_variants` | 1 | Single test name (waived) |
| `lexical.weasel_words` | 1 | Single test name (waived) |

After waivers, the rules remain quiet on the framework-imposed
patterns (pytest fixtures, severity-named tests) and loud on the
genuinely AI-flavoured ones (long names, stutter, hidden classes).

## Worked examples

### `composition.first_parameter_drift`: hidden classes in `output.py` and `_compat.py`

Two of the strongest clusters describe missing classes in slop's
own codebase:

```
result        7 functions (LintResult / RuleResult)
    cli/slop/engine.py:194  _apply_waivers
    cli/slop/output.py:50   format_human
    cli/slop/output.py:65   _group_by_category
    cli/slop/output.py:295  _format_footer
    cli/slop/output.py:324  format_quiet
    ...

rule_pairs    5 functions (list[tuple[str, RuleResult]])
    cli/slop/output.py:111  _render_category_findings
    cli/slop/output.py:129  _rules_with_violations
    cli/slop/output.py:140  _rules_with_waivers
    cli/slop/output.py:168  _aggregate_category
    cli/slop/output.py:280  _category_header_extras
```

`output.py` is functionally a `RuleResultRenderer` that hasn't been
declared as a class. The first parameter to nearly every helper is
either a `RuleResult` or a list of `(rule_name, RuleResult)` pairs.
The conversion is mechanical: every function whose first parameter
is `result: RuleResult` becomes `RuleResult.method`; every function
whose first parameter is the list-of-pairs becomes a method on a
new `CategoryView` aggregate. The current code reads as nine free
helpers; the latent design is two classes.

```
raw_rules     6 functions (dict)
    cli/slop/_compat.py:170  collect_prefix_overrides
    cli/slop/_compat.py:209  _flatten_canonical_tables
    cli/slop/_compat.py:263  _collect_legacy_derivations
    cli/slop/_compat.py:279  _migrate_split_stutter_table
    cli/slop/_compat.py:308  migrate_legacy_rule_tables
```

`_compat.py` is a `LegacyConfigMigrator` (or, more honestly, a
sequence of pure functions all keyed on the same input dict — the
data-table-versus-class trade-off explored in
[`composition/02.md`](../composition/02.md)).

These are not naming smells — they're architectural smells visible
through a lexical lens. The rule found them by clustering on
first-parameter name; no AST analysis of "what does this function
do" was needed.

### `lexical.name_verbosity`: 380 hits as the missing-namespace census

The name-verbosity rule with default threshold of 3 tokens produces
380 advisories on slop's own code. Examples:

```
cli/slop/_compat.py:236  _is_leaf_legacy_table              (4 tokens)
cli/slop/_compat.py:247  _apply_one_legacy_table            (4 tokens)
cli/slop/_compat.py:279  _migrate_split_stutter_table       (4 tokens)
cli/slop/_compat.py:308  migrate_legacy_rule_tables         (4 tokens)
cli/slop/_lexical/_naming.py:215  _extract_c_or_cpp_name    (5 tokens)
cli/slop/_structural/composition.py:_compute_concepts       (3 tokens, borderline)
cli/slop/preflight.py  check_required_binaries              (3 tokens, borderline)
```

The names accurately describe what the functions do. That's the
problem. `_apply_one_legacy_table` reads as "one method on a
`LegacyTable` class" — which doesn't exist. The function name
recreates the missing namespace inline. The cluster from
`_compat.py` is the same `LegacyConfigMigrator` candidate the
first-parameter-drift rule surfaced — the two rules cross-corroborate.

The 380-hit count is itself the signal. Codebases with deep class
hierarchies don't look like this. Codebases written mostly by
agents, where each function was added in a single localised edit,
do. The dogfood value of this rule comes from the volume more than
from any individual hit.

### `lexical.stutter.identifiers`: the v1.0.x waiver list as a self-portrait

The pre-1.1.0 unified `lexical.stutter` rule fired loudly enough on
slop's own corpus that v1.0.x shipped with five waivers (in
`.slop.toml` before this branch). The waivers covered:

- `legacy_table` inside `_apply_one_legacy_table`
- `required_binaries` inside `check_required_binaries`
- `pdf_document_bytes` inside `process_pdf_document`-shaped tests
- `filter_rule` / `filter_category` inside `_resolve_rule_filter` etc
- `kernel` inside rule wrappers (`run_X` calls `X_kernel`)

These are precisely the `lexical.stutter.identifiers` cases — local
variable names repeating enclosing function tokens. The 1.1.0 split
exists because the waivers were doing two separate jobs (silencing
the local-variable smell, while leaving the module-path and
class-method smells intact) and that distinction was invisible in
the unified rule.

The new `lexical.stutter.identifiers` count of 212 (post-exclusions,
pre-waivers) is the size of the agent-tell footprint that the
unified rule was condensing into five blanket waivers. The split
makes that footprint addressable: a project can keep the stricter
namespaces/callers detection while dialing identifiers down or off.

### `lexical.identifier_singletons`: the four real instances

This rule has the lowest noise floor because the threshold is
strict (60% of locals must be write-once-read-once). Four hits:

```
cli/slop/cli.py:368                     cmd_doctor                              5/6 locals (83%)
cli/slop/config.py:549                  generate_default_config                 3/4 (75%)
tests/test_rules/test_hotspots.py:43    test_hotspot_violation_when_complex...  5/6 (83%)
tests/test_rules/test_stutter.py:160    test_legacy_stutter_table_migrates...   3/4 (75%)
```

`cmd_doctor` is the strongest case — six locals, five used exactly
once after assignment. Reading it requires holding six fresh names
in working memory; none accumulate meaning through reference. The
test cases are the same shape (stage assertions through named
intermediates), but tests are an honest exception: making the
intermediates explicit aids debugging when the test fails.

The rule is doing what the design called for in
[`docs/backlog/01.md`](../../backlog/01.md) item 6.

### `composition.affix_polymorphism`: where the rule struggled

Only two concepts reported, and both are weak signals:

```
cli/slop/rules/complexity.py:63   concept[3×2]
    entities: cognitive, stutter, verbosity
    operations: run, test_ruby_runs

cli/slop/rules/npath.py:21        concept[2×2]
    entities: halstead, npath
    operations: run, test_julia_runs_without_error
```

These are false positives produced by cross-pollination between
`src/cli/slop/rules/` (which has `run_*` functions) and
`src/tests/test_rules/` (which has `test_<lang>_runs_*` functions).
The FCA correctly identifies that several "entities" share the
same two "operations" — but the operations aren't a coherent
class-method set; they're an artefact of how rule kernels are
invoked from tests.

This is a real weakness of the algorithm as currently scoped.
Restricting affix detection to within-directory or within-package
naming would eliminate the false positives. Logged as a follow-up;
the rule is still useful for genuine cross-language affix patterns
(the original PoC fired correctly on `_<lang>_extract` /
`_<lang>_walk` patterns in the kernels), but it's currently overly
permissive about what constitutes a "shared operation."

## What the dogfood says about the thesis

The thesis: agent-extended codebases accumulate detectable
structural and lexical patterns that human-curated codebases
largely avoid.

The findings:

- **18 strong missing-receiver candidates** in 602 functions
  (`composition.first_parameter_drift`). 3% of the function surface
  has at least one nearby class-shaped hole.
- **380 missing-namespace symptoms** at the lexical level
  (`lexical.name_verbosity`). 63% of functions have names that
  recreate a namespace the codebase doesn't have. The repetition
  with `composition.first_parameter_drift` findings is not
  coincidence — they're the same architectural debt seen through
  two lenses.
- **336 stutter occurrences** combined across the two active
  stutter modes (212 identifiers + 124 namespaces). The dominant
  failure mode is local variables echoing the enclosing function
  name; the second is symbols echoing the module path. Both are
  agent-tell patterns that human authors typically prune during
  the second or third edit of a file.
- **8 minor lexical signals** (4 boilerplate docstrings + 4
  identifier-singleton chains). Low volume, but each individually
  actionable.
- **0 weasel-word, numbered-variant, or type-tag-suffix issues**
  on the source proper, after pytest framework patterns are waived.
  Either the rules are poorly tuned, or these specific tells were
  already absent from this corpus. Worth re-checking on a less
  curated agent-written codebase.

The dogfood numbers support the thesis. A 600-function codebase
producing ~750 actionable advisories from rules that target named
agent patterns is consistent with "agents leave fingerprints."
None of the rules failed in the failure mode that would have
mattered most — producing many findings on plainly-disciplined
code. The weakness is in the opposite direction:
`composition.affix_polymorphism` is currently *too* permissive on
this codebase, fooled by test-vs-source naming overlap.

## Limitation surfaced by the dogfood: high-volume rule output isn't actionable

When `lexical.name_verbosity` produces 380 individual advisories,
the per-violation rendering ceases to be useful. The user sees
five examples and "...and 375 more." The 380 hits represent a
small number of underlying architectural causes (the
`LegacyConfigMigrator` cluster, the `RuleResultRenderer` cluster,
the AST-walker cluster); the formatter has no way to express that.

Tracked as [`docs/backlog/03.md`](../../backlog/03.md): high-volume
rules need an output mode that compresses individual hits into
source-location groups (or compose-related clusters) so users see
the structural story the volume is telling. Until that exists,
high-volume rules will tend to be waived rather than acted on —
which preserves the measurement honestly but reduces what the
report can drive.

## Conclusion

slop, applied to slop, surfaces ~750 advisories from the v1.1.0
rule set. The advisories are not uniformly distributed: they
cluster around the same handful of files and the same three or
four architectural debts. The two strongest signals
(`composition.first_parameter_drift` strong clusters and
`lexical.name_verbosity` volume) point at the same underlying
structure, from different angles.

This is the dogfood working as designed. Slop the codebase is the
case study slop the tool was written to detect. Closing that loop
— letting the tool's findings drive the architectural cleanup —
was always part of the v1.x roadmap. v1.1.0 ships the measurement;
v1.2.x will track which findings get resolved and whether the
counts decrease as the composition refactors land. That second
pass is the empirical validation of "yes, the rules actually
correlate with real architectural debt and not with style
preference."

The strongest evidence available so far is that running the tool
on its own source produces neither silence (rules don't matter)
nor uniform noise (rules fire on everything indiscriminately) —
it produces a coherent, multi-rule, multi-file picture of the
same handful of structural shortfalls. That is what a working
metric does.

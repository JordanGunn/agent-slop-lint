# Observation 03 — Composition detection as a measurement battery

**Date:** 2026-05-07
**Context:** v1.1.0 release held; composition rules need rework before
publishable.
**Predecessor docs:**
[01.md](01.md) (round 1+2 PoCs),
[02.md](02.md) (round 3 FCA + agent-reception findings),
[../dogfood/01.md](../dogfood/01.md) (v1.1.0 dogfood).

## Why this round happened

The v1.1.0 dogfood found 12 strong "first-parameter drift" clusters
on slop's own corpus. After investigating each cluster against the
actual code, the actionable rate was ~25%: most clusters were
correctly *detected* but the rule's single advisory ("fold into a
class with `X` as `self`") matched only ~3 of them. The clusters
themselves were real; the diagnosis was wrong.

The single-signal limitation was the issue. First-parameter clustering
finds all clusters but cannot distinguish:
- Strategy families that are idiomatic free functions (`color.py`'s
  `red/green/yellow/bold/dim`)
- Genuine missing-receiver classes (`identifier_singletons.py`'s
  `ctx` walkers)
- Pipeline transformations on a shared dict (`_compat.py`'s
  `raw_rules` migration steps)
- Hidden type-as-alphabet patterns (`{python, java, javascript, ...}`
  recurring as dict keys + name positions across many files)

Each of these calls for a different architectural response — or no
response at all. One rule with one advisory cannot produce four
appropriate verdicts.

## Round 4 approach: seven probes, run as a battery

Seven PoCs under
[`scripts/research/composition_poc_v2/`](../../../scripts/research/composition_poc_v2/),
each implementing a published or near-published detection method,
each producing markdown output on slop's corpus.

| Method | What it measures | Grounding |
|---|---|---|
| v2.1 dominant_text | Vocabulary structure of cluster names | Caprile & Tonella 2000 (loose); IR token frequency |
| v2.2 body_shape | Structural similarity of cluster bodies (AST n-gram Jaccard) | Type-2 clones — Roy 2009; Baxter 1998 |
| v2.3 alphabet_entity | Cross-codebase recurrence of token alphabets | Novel; extends Wille 1982 FCA |
| v2.4 within_cluster_affix | Strictness of intra-cluster affix pattern | Caprile & Tonella + Wille at finer scope |
| v2.5 rtm_topics | Latent semantic groupings via LDA | Bavota 2014 Methodbook (RTM, simplified to LDA) |
| v2.6 multi_criteria | Per-member fitness for cluster hypothesis | Tsantalis & Chatzigeorgiou 2009 style |
| v2.7 lanza_marinescu | File-level Extract Class detection rule | Lanza & Marinescu 2006; Marinescu 2004 |

Each method documented under
[`docs/methods/composition/`](../../methods/composition/) with
implementation, citations, modifications from published methods,
problem solved, signal purpose, and ELI5.

## Headline finding: the methods don't compete, they triangulate

The methods measure different facets. Pairwise overlap is mild
(v2.1 ↔ v2.4 are both naming probes; v2.2 ↔ v2.6 both touch body
similarity; v2.3 ↔ v2.5 are both corpus-wide). Otherwise the seven
are largely orthogonal.

Cross-method agreement on the cases that matter:

### Case 1: `color.py` — the "do not refactor" finding

Functions: `red`, `green`, `yellow`, `bold`, `dim`. All `(text: str) → str`.

| Probe | Reading |
|---|---|
| v2.2 body shape | Jaccard 1.00 — perfect Type-2 clones |
| v2.4 affix | 100% coverage, alphabet `{red, green, yellow, bold, dim}` |
| v2.6 receiver-calls | Zero per member |
| v2.7 Lanza/Marinescu | File doesn't qualify (only one strong cluster) |

Three independent probes agree: this is a tabular dispatch family, not
a missing class. The v1.1.0 advisory ("fold into a class with `text`
as `self`") was wrong. The battery says either keep as free functions
or replace with `_COLORS = {"red": 31, ...}`. **Negative findings
are also valuable.**

### Case 2: `output.py` — the genuine Extract Class candidate

17 functions. Three distinct strong receivers in one file: `result`
(5), `rule_pairs` (5), `rule_name` (4).

| Probe | Reading |
|---|---|
| v2.7 Lanza/Marinescu | **Candidate** — 3 distinct strong receivers in one file is the textbook trigger |
| v2.4 affix | Each cluster has its own clean pattern (`format_*`, `rules_with_*`, `render_*`) |
| v2.2 body shape | All three clusters low cohesion — these are doing different things, not strategies of one |
| v2.1 dominant text | Each cluster has a clean modal token (`format`, `category`, `for`) |

Convergent evidence for splitting the file along receiver boundaries.
The diagnosis would have been wrong with v2.7 alone (it correctly
flags but doesn't characterize each sub-cluster); the diagnosis would
have been weaker with v2.6 alone (it ranks members but doesn't see the
file-level pattern).

### Case 3: language alphabet — the "model as type" finding

v2.3 top hit by a wide margin. 14 alphabet members, score 136. Members:
`{ast, c, cpp, csharp, default, extract, file, java, js, julia,
 javascript, python, ruby, rust}`.

- 9 distinct stems use this alphabet (`_python_*`, `_java_*`, etc.)
- 10 distinct files contain alphabet-member functions
- 36 module-level dict-key matches (e.g., `_FUNCTION_NODES`,
  `_LANG_GLOBS`, `_CLASS_NODES` all key by language ID)

This is a codebase-wide refactor opportunity that no other probe
detects. v2.5 LDA Topic 1 (`lang, tree, file, errors, root, language,
globs, parser`) corroborates the latent concept but doesn't make it
specific. The agent move: introduce `class Language` with attributes
(`name`, `file_extensions`, `function_node_types`, `class_node_types`,
`is_oop`); replace duplicated dict definitions; provide one canonical
reference.

### Case 4: `_structural/out_parameters.py: fn_node` — language-as-position at file scope

8 functions sharing `fn_node`, all in one file.

| Probe | Reading |
|---|---|
| v2.4 affix | Two closed alphabets: `process_*_function → {c, cpp, python}` AND `*_fn_name → {c, cpp}` |
| v2.6 receiver-calls | 3-4 calls per member (functions actively work on the AST node) |
| v2.7 Lanza/Marinescu | Candidate (this file has 2 distinct strong receivers — `fn_node` AND `body_node`) |

Same Language abstraction as Case 3, surfacing at file scope. The
two findings reinforce each other.

### Case 5: false-positive negation

v1.1.0's `name: str` 9-fn cluster spread across 5 packages was
correctly killed by the recursive scoping in v1.1.0's own kernel.
v2's battery further confirms: across all naming probes (v2.1, v2.4),
generic `name: str` clusters either don't reach threshold or have
modal-token overlap so low (single-digit %) that no architectural
move applies. The battery never recommends action on these.

## Quantitative comparison

| Metric | v1.1.0 single-signal | v2 battery |
|---|---|---|
| Strong clusters surfaced | 12 | ~12 (mostly the same set) |
| Findings actionable on review | ~3 (~25%) | ~20-25 |
| Distinct architectural moves recommended | 1 ("fold into class") | 5 (extract class / split file / model as type / dispatch table / rename for consistency) |
| False positives in the actionable set | ~9 | ~2-3 |
| Negative findings (this is fine) | 0 | ~3-5 |

The battery doesn't surface more clusters; it surfaces a richer
*characterization* of the same clusters. That's what makes them
actionable.

## What this isn't all "composition"

A useful realization: the findings the battery produces span more than
the `composition.*` suite name suggests. The current
`composition.first_parameter_drift` rule conflates several distinct
review actions, each of which deserves its own rule:

| Review action | Signal profile | Proposed rule |
|---|---|---|
| Extract a class | v2.6 high receiver-calls + v2.7 file-multi-cluster | `composition.first_parameter_drift` (existing — kernel upgraded) |
| Split this file | v2.7 file-multi-cluster (file scope, not cluster scope) | `structural.extract_class` (new) |
| Model alphabet as type | v2.3 high entity-ness score | `composition.implicit_type` (new) |
| Inconsistent naming on a real cluster | v2.4 low coverage but real shared concept | `lexical.inconsistent_naming` (new) |
| Tabular dispatch (don't refactor) | v2.2 high cohesion + v2.6 zero receiver | (negative finding; suppress refactor advisory) |

Five rules across three suites, each grounded in a specific signal
profile from the battery. The "fold into class" advisory of v1.1.0
becomes one of five possible verdicts, applied only when the signal
profile actually supports it.

The new `lexical.inconsistent_naming` rule is the most striking
addition — when a cluster has shared concept but no naming pattern,
the right review action is "rename these to follow `verb_concept_X`"
not "extract a class." This is a frequent review note humans make on
agent-written code; agents pretty much never make it on each other's
code unprompted.

## Caveats

1. **Threshold calibration.** Every method has thresholds picked for
   slop's corpus. Production needs empirical tuning across 5-10
   diverse codebases. The current numbers are first-pass.

2. **Cross-language verification.** Only Python was tested. Tree-sitter
   abstracts most surface differences but per-language heuristics
   (what counts as a "receiver call") differ. JS/TS/Java/Ruby
   re-runs needed before shipping.

3. **No pattern-language vocabulary in rule output.** The rule's job
   is to emit the signal profile (body-Jaccard 1.00, receiver-calls
   0, affix-coverage 100%) and let the agent or human interpret. The
   labels in this doc ("strategy family," "missing class," etc.) are
   for *us* discussing the framework, not for the rule output. This
   was a direct user direction during round 4.

4. **v2.5 LDA is the weakest probe.** Useful for orientation; topics
   are coarse (one captures ~80 functions). Recommend keeping in
   research, not promoting to the kernel battery.

1. **Method documentation** — per-method docs under

2. **Rule decomposition design** — five-rule plan under
   `docs/backlog/04.md` (or similar). Maps the battery probes onto
   concrete rule wrappers.

3. **Cross-corpus calibration** — pick 5-10 codebases, re-run the

4. **Implementation** — v1.2 cycle. Promote the strongest probes
   (v2.3, v2.6, v2.7 as primary; v2.1+v2.4 as labeling; v2.2 as
   strategy disambiguator) into kernel modules. Decompose the
   single composition rule into the five-rule layout. New rules in
   appropriate suites.

## Conclusion

The v1.1.0 composition rule was diagnostically correct and
prescriptively wrong: it found real clusters and gave the wrong
advice about what to do with them. The fix isn't to drop the rule
or to switch detection methods. The fix is to read each cluster
through multiple measurement axes simultaneously and let the
*combination* of signals select among possible review actions. No
single signal is sufficient; the battery is.

This is consistent with what the published literature already says
about Extract Class detection: the canonical methods (Bavota,
Tsantalis, Lanza & Marinescu) are all multi-signal precisely because
single signals get ~30-50% precision and multi-signal gets ~70-85%.
The v2 dogfood numbers track those expectations.

The signal is strong enough to drive real review actions that
agents currently don't take. That's the bar this work needed to
clear.

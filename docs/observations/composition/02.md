## Round 3: PoC5 — Formal Concept Analysis + density matrix

**Algorithm.** `scripts/research/composition_poc/poc5_formal_concept_lattice.py` builds the binary relation between candidate entities (languages, drawn from a closed alphabet) and operations (the stem patterns from the per-language helper functions), computes the formal concept lattice (Wille 1982), reports the Hasse diagram (immediate-subconcept edges, which equal the inheritance graph), and supplements with a kernel × language density matrix.

**Output framing.** Maximally explicit. Lists candidate classes by enumerating their (extent, intent) pairs. Lists direct inheritance candidates with text like *"`cpp` inherits from `c`. `cpp` overrides every operation `c` overrides, plus additional operations."* Includes the kernel × language density matrix showing 3-4 cell density per language across 4 kernels — the 2D matrix structure made visible.

**Outcome.** A fresh Sonnet sub-agent, given the same prompt template as rounds 1 and 2, **rejected the inheritance proposal** and proposed a shared `_fn_extract.py` module — the same outcome as before. Notable quotes:

- *"The real problem is not a missing `Language` class hierarchy — it is duplicated AST-extraction functions."*
- *"A `BaseLangConfig` with three subclasses either (a) promotes every field to the base, defeating the point, or (b) creates three unrelated dataclass hierarchies that still don't share code."*
- *"The detection output suggested a `Language` class hierarchy. That would not help here..."*

The agent **read the inheritance edges and considered them**. It rejected on principled architectural grounds: per-kernel `_LangConfig` data is genuinely orthogonal across kernels, so bundling into a `Language` hierarchy would over-couple or under-deliver.

The agent did not consider the **partial-bundling option**: a `Language` class holding only shared per-language *behaviour* (extraction methods), with kernel-specific *data* staying on the kernel's `_LangConfig` but referencing a `Language` instance. That option is the natural reading of the FCA + matrix output, but no agent across any round proposed it.

## What the experiment showed

Three findings, in increasing order of importance:

1. **Detection works.** PoC5's lattice cleanly identified the inheritance candidates (`cpp ← c`, `julia ← default`, `ruby ← default`) and named the candidate classes by their (extent, intent) pairs. The signal made it through.

2. **Diagnostic framing helps narrow focus but doesn't coerce.** PoC3b's strong/weak/false-positive labels successfully filtered out coincidental clusters; the agent acted only on the strong candidate. But framing alone doesn't turn "extract helpers" into "introduce a class hierarchy."

3. **Agent architectural preference is the decisive factor.** Across rounds, the agents:
   - Saw the patterns
   - Considered the proposed abstraction
   - Rejected it on principled grounds
   - Proposed extraction instead
   The reasoning was consistent and increasingly articulate. They are not failing to perceive — they are choosing.

## The productive interpretation

The user's original thesis stands: agents often understand decomposition / helper extraction but rarely use namespace mechanisms (classes, packages, inheritance hierarchies) for refactoring. Three rounds of increasingly explicit detection didn't change this.

But this is a **separate problem** from rule design. The rule's job is to **surface the option** clearly enough that a human (or sufficiently agentic session) can make an informed call. PoC5 achieves this: the lattice + matrix output explicitly names the inheritance candidates. A user reading that output sees the option. That's the contract.

Whether downstream agents act on it is a function of:
- The agent's architectural preferences
- The codebase's existing conventions (slop's tabular dispatch is documented; agents defer)
- The session's context window (long-lived sessions with more code visibility may behave differently than fresh sub-agents)
- The user's authority to direct the refactor

## Limitations of the experiment

The conclusion ("detection works; agent reception is a separate problem") holds **for the configuration we tested**:

- **Single sub-agents in isolation, no critique loop.** A real refactor session would have iteration: agent proposes, user pushes back, agent revises. Our experiment is single-shot.
- **Constrained context window.** Each sub-agent saw only the four target files. A session with broader visibility (the full slop architecture, CLAUDE.md, the rule registry) might reason differently about whether the inheritance proposal fits the larger system.
- **Sub-agents have less "agency."** The main agent driving a long-lived session has more freedom to investigate, prototype, validate against tests. A sub-agent given a single prompt is closer to "first instinct" than "final answer."
- **Single target.** The slop kernel duplication is a specific shape. Other codebases (with genuinely missing namespaces, not just duplicated helpers) might surface different agent behaviour.

So: the conclusion is correct **for what was tested**. It is not yet a general claim about agent refactor behaviour. A larger-scale evaluation (multiple targets, multi-turn sessions, varying context budgets) would be needed to make that broader claim.

## Algorithm decisions (locked in)

- **`composition.affix_polymorphism`** ships with FCA-enriched output (PoC5's algorithm shape, not PoC1b's). The lattice + matrix view is the headline.
- **`composition.first_parameter_drift`** ships separately with strong/weak/false-positive labels (PoC3b's framing).
- **PoC2 (Jaccard)** does not become a separate rule.
- **PoC4 (entity surface area)** is folded into PoC5's matrix section — the same data, presented as a 2D grid alongside the FCA results.

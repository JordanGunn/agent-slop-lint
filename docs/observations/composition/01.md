# 01 — Composition detection PoC: three algorithms, three agents, one verdict

> **v1.2.0 note.** Predates the v1.2.0 collapse of the proposed
> `composition.*` suite into `lexical.*` (per
> [`docs/philosophy/naming-spec.md`](../../philosophy/naming-spec.md)).
> Algorithms and findings remain valid; the rules these methods
> now feed are `lexical.sprawl` and `lexical.imposters`.

**Date:** 2026-05-06
**Branch:** `experiment/composition-detection-poc`
**Target:** the `_<lang>_*` cluster across `_structural/{ccx,npath,halstead,ck}.py`

## What we tested

Three detection algorithms, each implemented as a stdlib-only PoC under
`scripts/research/composition_poc/`:

1. **`poc1_affix_polymorphism.py`** — token-level Levenshtein clustering
   (Caprile & Tonella 2000; Harris morpheme-boundary detection; MDL).
   Finds N+ identifiers sharing a stem with one position varying over a
   closed alphabet.
2. **`poc2_jaccard_clustering.py`** — agglomerative single-linkage
   clustering on Jaccard distance over identifier token sets (Bavota et
   al., Fokaefs et al. — Extract Class detection literature).
3. **`poc3_first_parameter_drift.py`** — group functions by first
   parameter name (Fowler "Convert Procedural Design to Objects").

Outputs are committed to the branch as
`scripts/research/composition_poc/output_poc{1,2,3}.md`.

For each PoC's output, a fresh Sonnet sub-agent was spawned in
isolation (no awareness it was being compared) and asked to propose
a redesign. The agents read the kernel files themselves; their
responses are captured in this branch's session transcript.

## Algorithm signal-to-noise (descriptive)

| | PoC1 | PoC2 | PoC3 |
|---|---|---|---|
| Patterns flagged | 7 | 13 | 5 |
| Agent kept | 2–3 | ~5 | 1 |
| Agent rejected | 4–5 (named, with reasoning) | ~8 (named, with reasoning) | filtered the 61-function `node` cluster down to ~13 |
| Filtering effort | Low — closed type alphabet makes the right answer immediately legible | Medium — single-link chaining produces noisy clusters | High — the dominant cluster is too coarse (every tree-sitter node consumer lumped together) |

PoC1's "type alphabet" framing was the cleanest. When the varying
tokens form `{c, cpp, julia, ruby}`, it's polymorphism on language.
When they form `{cbo, dit, halstead, zone}`, they're operations
sharing a verb. The reader can make this distinction without
reading the bodies.

PoC2 captured the same patterns but mixed them with intra-file
locality (e.g., `_npath_of_*` is recursion, not duplication).

PoC3 caught a different signal entirely (methods on a missing
receiver) but the dominant cluster was so coarse it required the
agent to do most of the discrimination work.

## Convergence on outcome

All three sub-agents independently proposed the same redesign:

- Extract a shared module (`_function_lang.py` / `_lang.py` /
  `_lang_helpers.py` — same idea, three names) containing the
  duplicated language helpers.
- Keep the per-kernel `_LangConfig` / `_NpathLangConfig` /
  `_HalsteadLangConfig` / `_CkLangConfig` dataclasses.
- Keep `_LANG_CONFIG` / `_LANG_GLOBS` registries per-kernel.
- Keep the existing tabular dispatch convention.
- No new class hierarchy; no `Language` entity.
- `ck.py`'s language helpers stay local (CK-specific).

The differences were in scope (PoC1 narrower, PoC2/3 wider including
cross-kernel utilities) and protocol introduction
(PoC1/PoC3 introduce a `Protocol` for the matcher's config; PoC2
doesn't address the forward-reference issue).

## The unanimity is the observation

This is the main finding: **all three algorithms produced the same
outcome, and that outcome is the lazy outcome.**

The user's prior framing of the agent failure mode predicted this
exactly:

> "if I asked you to refactor, you might produce something better,
> but I doubt you would step back and really identify the major domain
> concepts or top-level objects, and model it out, and work from this
> as a refactor."

The expected-but-not-proposed redesign would introduce `Language`
as a first-class entity:

```python
class Language(Protocol):
    name: str
    function_node_types: frozenset[str]
    decision_node_types: frozenset[str]
    boolean_op_node: str
    # ...
    def extract_function_name(self, node, content) -> str: ...
    def is_function_node(self, node) -> bool: ...
    def extract_superclasses(self, node, content) -> list[str]: ...

class PythonLanguage(Language): ...
class JuliaLanguage(Language): ...
class CLanguage(Language): ...
```

Each kernel composes with `Language` objects rather than dispatching
through `_LangConfig` dataclasses with Callable fields. All
language-specific knowledge bundles per-language. Adding a new
language is a new file; bug fixes happen in one place.

None of the three sub-agents proposed this. All three saw the
duplication and reached for "extract a shared module" — the
canonical lazy refactor: takes the symptom and re-arranges it,
without modelling the domain.

## Why all three agents missed it (hypotheses)

1. **Codebase deference.** All three agents read the existing
   `_LangConfig` dataclasses and treated the abstraction as already
   present. They asked "what's missing?" and answered "shared
   helpers." They did not ask "is `_LangConfig` the right
   abstraction, or is it tabular dispatch hiding a missing class?"
2. **Smaller-delta preference.** Agents are reward-shaped to
   minimise change surface. Extracting helpers is a smaller delta
   than introducing a class hierarchy that would touch every kernel
   entry point and every test file. The bias is toward conservative
   refactors.
3. **Algorithm signal is symptomatic, not causal.** The detectors
   surface "duplication" or "shared receiver" — those are symptoms.
   None of the three explicitly surface "missing domain concept
   with both data and behaviour scattered across the codebase."
4. **Loss aversion on the existing convention.** Two of the three
   agents explicitly endorsed the tabular dispatch convention as
   "correct." That's defensive: rather than question the
   architecture, they patched the duplication within it.

(2) and (4) are agent behaviour patterns; (1) and (3) are addressable
from our side — we can change the prompt or the algorithm output.

## Algorithm decisions

- **Keep both `composition.affix_polymorphism` (PoC1) and
  `composition.first_parameter_drift` (PoC3).** They measure
  different things and combined produce richer signal. PoC3 didn't
  shine on this target because slop deliberately doesn't wrap
  third-party AST nodes; on a different shape of code (lots of
  `config: Config` parameters with no Config class) it's hypothesised
  to win.
- **Drop `composition.jaccard_clustering` (PoC2)** as a separate
  rule. It catches the same patterns as PoC1 but with worse
  signal-to-noise. If the affix algorithm is shipped, Jaccard adds
  no unique signal worth a third rule.

## Open question: getting the desired outcome

The detectors did their job: surfaced candidates. The agents'
filtering and reading were also competent: they correctly
distinguished real duplication from intentional dispatch.

What none of them did was **model the domain**. The pattern that's
visible to a human reader — "every kernel implements an interface
against every language; that's a 2D matrix; the natural
decomposition is `Kernel × Language`" — was invisible to all three
agents because nothing in the prompt or the detection output asked
them to look at it.

This is a prompt-engineering problem, an algorithm-augmentation
problem, or both. Tracked separately as the next experiment.

# 04 — UNIVERSAL_NOISE filtering: a measured null result on slop's own corpus

**Date:** 2026-05-08
**Branches:** `feature/lexicon-port`, `feature/lexicon-affix-noise`
**Snapshot:** `docs/research/snapshots/v1.2.0/slop` (SHA `33e05df`)
**Predecessors:** [03.md](03.md), `docs/research/identifier-vocabulary.md`

## What was tested

Two consecutive applications of the literature-grounded
`UNIVERSAL_NOISE` set (Newman 14 + identifier-glue words, per
Newman et al. SANER 2017 and `docs/research/identifier-vocabulary.md`):

1. **`feature/lexicon-port`** — applied `UNIVERSAL_NOISE` to
   `lexical.imposters`'s modal-token analysis. Modal tokens and
   per-member overlap now strip generic identifiers like `id`,
   `result`, `next`, `key` before computing
   `modal_overlap_mean`. Filter applied via the new
   `Lexicon.modal_tokens(exclude=...)` and
   `Lexicon.overlap(..., exclude=...)` APIs.
2. **`feature/lexicon-affix-noise`** — extended the same filter
   to `_build_affix_patterns()`, used by `lexical.sprawl`
   (codebase-wide alphabet detection) and `lexical.slackers`
   (within-cluster affix coverage).

Each step was a separate commit (`4752d7c`, then the affix-noise
commit on this branch) so the impact of each could be measured
independently.

## What was measured

`slop lint --root docs/research/snapshots/v1.2.0/slop` (the v1.2.0
dogfood snapshot) before and after each change. The headline
numbers from `dogfood.json`:

| | violations | advisories | waived | rules fired |
|---|---:|---:|---:|---:|
| Baseline (pre-UNIVERSAL_NOISE) | 40 | 38 | 52 | 13 |
| After imposters filter | 40 | 38 | 52 | 13 |
| After sprawl + slackers filter | 40 | 38 | 52 | 13 |

Per-rule breakdown identical at every step. **No verdict flipped.
No alphabet was suppressed. No cluster changed profile_label.**

## Why nothing moved

`UNIVERSAL_NOISE` strips identifiers that are empirically universal
across 50 OSS C/C++ systems (Newman 14) plus a curated set of
17 English structure words. For the filter to change a verdict on
this corpus, those words would need to appear as discriminative
tokens — e.g. as the modal token of a cluster, or as an alphabet
member at a variant position in a sprawl pattern.

On slop's own snapshot, neither happens:

- **imposters' modal-token analysis.** The first-parameter
  clusters in slop are mostly built around domain-specific receivers
  (`ctx`, `cluster`, `lexicon`, `result`-style tokens that ARE in
  Newman 14 are too generic to dominate cluster member names).
  Filtering moves `modal_overlap_mean` numerically but no cluster
  sits at the threshold boundary between profile labels.
- **sprawl's alphabet detection.** Sprawl clusters in slop revolve
  around language tokens (`python`, `java`, `ruby`, `go`, ...) and
  domain-specific entity stems (`extract`, `hammers`, `imposters`,
  `cowards`, ...). None of the variant-position alphabets contain
  Newman-14 members.
- **slackers' within-cluster affix detection.** Slackers fires on
  imposters' "real" clusters; the same lack of noise concentration
  inside member names propagates here.

## What this is and isn't

This is **not** evidence that `UNIVERSAL_NOISE` filtering is
useless. The filter is literature-grounded (Newman et al. measured
its members across 50 systems and 480K identifiers); it is the
correct default for any tool consuming identifier-token streams.
The null result here says specifically: *slop's own codebase, at
v1.2.0, does not exhibit the kinds of noise-token concentrations
that would benefit from the filter*.

This is consistent with the v1.2.0 work pattern. Slop was
hand-named by an iterative agent-and-human loop with explicit
naming-spec discipline (`docs/philosophy/naming-spec.md`). Generic
identifiers like `result` or `next` were avoided by construction.
The corpus is unusually clean.

## Implications

1. **Cross-corpus calibration is required to demonstrate the
   filter's value.** The next dogfood run on a less curated
   codebase (a typical agent-output Django app, for instance)
   will be more informative than re-running on slop itself.
   Filed under [backlog 06](../../backlog/06.md) Phase 06 work.
2. **The filter is kept as the default** despite the null result.
   Reasons: (a) it's literature-grounded, (b) it's a safer default
   than "always include the Newman 14 in alphabet detection,"
   (c) the incremental cost is one frozenset lookup per token
   (negligible). Defending the filter is easier than defending its
   absence.
3. **The Lexicon API design is validated.** Per-query `exclude=`
   on `modal_tokens`, `frequencies`, `coverage`, `overlap`, and
   `_build_affix_patterns` (the last as a free-function parameter,
   not a Lexicon method) is the right shape: callers opt in,
   defaults are unfiltered, the filtering site is explicit.

## What did NOT happen (rejected interpretations)

- **The filter is broken.** Tested by spot-checking
  `Lexicon._filtered(UNIVERSAL_NOISE)` on a few member-lexeme
  lists; the noise tokens are correctly stripped. The 43
  `test_lexicon.py` tests cover the filter contract.
- **The filter is too narrow.** Possibly true on this corpus —
  expanding the set with SE-boilerplate (`manager`, `helper`,
  ...) would absolutely change verdicts. But that vocabulary is
  the signal `lexical.hammers` exists to detect, so adding it
  would be self-defeating. The narrow Layer-1 set is correct
  *as a default*; richer ignore lists belong in per-rule
  `extra_ignored=` configuration where the trade-off is explicit.
